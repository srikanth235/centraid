Path 1 · use it

From clone to a living vault.

Centraid is pre-release and installs from source — one clone, one bun install, one command to open the shell. This page is the whole first day: get it running, put an app on it, bring some of your real data in, and put it in your pocket. When something here makes you ask why, the understand track has the answer.

§01 · install

Clone, install, open.

You need two runtimes: Bun ≥ 1.3 (the package manager and dev runner) and Node ≥ 24, since it ships node:sqlite built in. macOS, Linux, and Windows all work.

terminal
# everything — shell, gateway, apps — comes from one repo
$ git clone https://github.com/srikanth235/centraid
$ cd centraid && bun install
$ bun run dev:desktop

dev:desktop opens the Electron shell with a gateway embedded in it — no separate server to run, no account to create. A per-launch bearer token guards the local HTTP surface; it lives in your OS keychain.

§02 · first open

A name, a color, a vault.

First launch asks exactly two things — what to call you, and an accent color. Behind that one screen the gateway mints your first vault: a directory on your disk holding vault.db (the canonical record of your life, on one shared ontology) and journal.db (the append-only audit trail). Every app you install, every automation you run, every agent you talk to works against that pair.

The shell lands on Home: your apps as a grid, a sidebar with your profile and a vault switcher, and four destinations up top — Discover, Assistant, Insights, Settings. You can create more vaults later (one per household member, one for an estate, one for testing) — each is a fully separate world.

§03 · your first app

Clone a blueprint, sign its grant.

Open Discover. Eight blueprint apps ship in the box — Docs, Photos, Notes, People, Locker, Tally, Agenda, Tasks — plus a shelf of automation templates. Cloning one is a single click; it lands on Home and opens like any app.

The part worth noticing: an app doesn't get your data by existing. Its manifest declares exactly which vault tables it wants to read and which typed commands it wants to call, and that declaration becomes a grant you can inspect and revoke. Install is not consent — the grant is.

Try Notes first: type @ in a note and link a person, a document, a task — cross-references are one mechanism everywhere. Apps § the blueprints →

§04 · talk to it

Ask across everything at once.

The Assistant tab is a chat over your whole vault — not one app's data, all of it. It reads with plain SQL over the ontology (receipted, row-capped) and writes only through the same typed commands your apps use. Ask it "what did I spend on the car this year, and where are the receipts?" and it joins finance rows to attached documents because they live in one model.

Inside every app there's a smaller version of the same idea — an Ask box wired to that app's declared queries and actions. Saying it and tapping it run the same handler.

assistantyour vault
you when did I last see Priya, and do I owe her anything?
assistant ⚙ vault_sql → people ⋈ interactions ⋈ tally Dinner on June 14th — three weeks ago. And yes: ₹1,840 from the Goa trip, split in Tally. Want me to draft a settle-up?

§05 · bring your data

Drop a file, review, publish.

Settings → Import takes the files other platforms let you leave with: .ics calendars, .vcf contacts, .mbox mail archives, CSVs, and zips of all of the above. Nothing lands silently — every import becomes a staged batch you review as a diff (create / update / skip, duplicates already resolved against what the vault knows) before you publish it in one transaction, with per-row provenance.

For live sources, automation templates ship for Gmail, Google Calendar, Google Contacts, and GitHub: you bring your own OAuth client, the broker keeps the tokens sealed — handler code never sees a secret — and each sync stages rows through the same review door. Connection health lives in Settings → Import; the guided connect wizard is still landing in the UI, so expect this corner to be the most beta of the lot.

§06 · pair your phone

One QR code, no sync service.

Settings → Phone → Connect a phone shows a QR code — a one-time pairing ticket that expires in minutes. Scan it with the Centraid mobile companion and the two devices hold a direct, encrypted peer-to-peer line (iroh QUIC) from then on: same apps, same data, live updates, no relay account, no cloud copy. Revoke a device any time from the same screen.

Being pre-release, the companion builds from source too: bun run dev:mobile runs the Expo project in apps/mobile (a dev build — the native tunnel module means Expo Go won't do).

§07 · go always-on

A gateway that outlives the laptop lid.

The desktop shell embeds a gateway, but automations only fire while it's open. The long-term shape is one always-on gateway — a spare machine, a home server — with every device, desktop included, as a client of it.

on the always-on machine
$ bun run build
$ centraid-gateway serve --data-dir ~/centraid-data --host 127.0.0.1 --port 8765
# admin from the same shell — pairing, devices, vaults, keys
$ centraid-gateway pair --data-dir ~/centraid-data --vault Local
$ centraid-gateway devices list --data-dir ~/centraid-data

Administration is deliberately shell-only — vault create/delete, device enrollment, and key custody never ride HTTP. Devices § topology →

§08 · back up the keys

Two files are the vault. One key seals it.

Backing up a vault is copying its directory — vault.db and journal.db travel as ordinary files. But secrets (passwords in Locker, OAuth tokens, card numbers) are sealed: encrypted at rest under a key that deliberately lives outside the vault directory, so a copied backup can't reveal them. That key is yours to custody:

key custody · receipted, shell-only
$ centraid-gateway key status --data-dir ~/centraid-data --vault Local
$ centraid-gateway key export --data-dir ~/centraid-data --vault Local --out seal-key.backup
# store seal-key.backup somewhere that isn't the same disk

Lose the vault, keep the key → restore and reveal. Lose the key → the sealed cells are gone; everything unsealed remains. Data § sealed columns → This is the manual version — automated, offsite, provider-backed backup with its own recovery kit is its own chapter →.
