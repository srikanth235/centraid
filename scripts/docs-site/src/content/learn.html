Learn it · the foundations

Nothing here is invented. That's the point.

Centraid is composed from ideas that already earned their keep — W3C vocabularies, IETF protocols, a database that is just a file, cryptographic keys as identity. This page is ten short primers, one per idea: the problem it solves, where it came from, and how it shows up in Centraid. No prior knowledge assumed, no information overload — just enough of each to read the rest of the docs with confidence.

§01 · the ontology idea

Agree on the nouns, once.

Every app you've ever used has quietly invented its own idea of "a person" and "an event." Your calendar, your budgeting app and your photo library each keep a private definition, so a question that spans them — "who did I spend money on and see in person this month?" — has no single place to ask. That's the silo disease: not missing data, but data that can't recognize itself across apps.

An ontology is the fix, and the word is less exotic than it sounds: it's a formal agreement about which things exist and how they relate to each other, written down once instead of reinvented per app. Centraid's ontology declares nine load-bearing types — party · place · event · account · transaction · content_item · activity · observation · link — plus SKOS concepts for vocabulary, all living in one canonical core.

Hard Rule #2 is "Extend, don't fork." A domain that needs a workout doesn't invent a second activity table — it holds a UNIQUE NOT NULL foreign key back to the core row it specializes, like health.workout.activity_id. Every row's identity is a UUIDv7: an immutable, meaningless key, so merges just join two rows on a core.link same-as relation instead of renumbering anything. And the apps you already know — Gmail, Drive, Photos — are treated as evidence of what a domain needs, never as the authority on how to model it: "Gmail stars ≠ Drive stars ≠ Photos favorites is a fragmentation artifact of separate databases, not a design to copy."

§02 · vocabularies

Naming things is its own hard problem.

Categories, tags and folders drift apart in every app that grows past its first year — one app's "Groceries" is another's "Food & Dining," and neither knows about the other. Fixing identity (the previous section) doesn't fix naming, which needs its own answer.

The prior art is SKOS, a W3C standard for controlled vocabularies: it defines concept schemes, concepts, preferred and alternate labels, and broader/narrower hierarchies for organizing terms. Centraid's tables mirror SKOS names literally — core_concept.pref_label is skos:prefLabel, core_concept.broader_concept_id is skos:broader — with one simplification: hierarchy is single-parent, a plain foreign key rather than a graph of RDF triples.

The payoff is that "Groceries" is one node — the budget, the agent and the export all cite the same row, so a rename touches a label, never an identity. SKOS concepts are the sole classification mechanism across Centraid: one flags-scheme tag replaced every per-app favorite boolean, and Docs folders are concepts too, not a separate filesystem-style hierarchy.

§03 · borrowed shapes

You don't invent a calendar event in 2026.

Decades of standards already encode the hard-won details of contacts, calendars, money and health data — the RSVP states, the reconciliation ids, the units of measure. Centraid's ontology doc footers its sources for exactly this reason: vCard · schema.org · RFC 5545 / JSCalendar · ISO 20022 / OFX · FHIR / Open mHealth · W3C PROV · ODRL / DPV · SKOS · MCP.

Standard What it settled Where it lives in Centraid
vCard 4.0 Name fields, contact channels, group membership party_identifier.scheme mirrors vCard EMAIL/TEL/URL; the label column is vCard TYPE (home/work/cell)
iCalendar RFC 5545 + JSCalendar The attendee RSVP state machine attendee.partstat is the PARTSTAT machine (needs‑action/accepted/declined/tentative); event.ical_uid survives round-trips to external calendars
schema.org A shared shape for people, places and creative work party.kind mirrors the Person/Organization split; content_item follows schema.org/CreativeWork
FHIR + Open mHealth + UCUM One shape for any measurement, clinical or not observation follows FHIR Observation, with UCUM units on value_num — "the same shape serves heart rates and water meters"
ISO 20022 / OFX Idempotent bank imports transaction.external_id is the OFX FITID / provider id, so re-importing a statement never double-posts
Double-entry & envelope budgeting (YNAB-style) Splits that reconcile, budgets that don't lie txn_split rows must sum to the parent amount; budget progress is a projection over core.transaction, never a stored balance

Some shapes get reused past their home domain — RRULE from RFC 5545 is stored, not just for calendar events, as the recurrence format for money cadences and medication schedules alike; Centraid never claims a full recurrence-expansion engine, only a shared way to write "every 2 weeks" down.

Borrowing shapes keeps round-trips honest — an ical_uid survives export and re-import unchanged — and it keeps divergences deliberate: an employment claim isn't a card field, because a job history needs provenance a vCard can't carry, so it's a core.link instead.

§04 · sqlite

A database you can put in your pocket.

Picture "a database" and you probably picture a server somewhere — something you rent, log into, and hope stays up. SQLite inverts that: it's a library your program links against, not a service you talk to over a network. The database is an ordinary file on disk. No server, no daemon, nothing to rent — which is exactly why it's widely described as the most deployed database engine in the world, quietly running inside phones, browsers and appliances everywhere.

That inversion is the sovereignty story. A vault is a directory of SQLite files on a machine you control: vault.db holds every schema of the ontology — the whole model, in one ACID boundary, so a single transaction can commit provenance, core rows and domain rows together with foreign keys engine-enforced. Alongside it, journal.db is the append-only audit stream — receipts, provenance, command invocations — kept separate so it can grow fast without ever touching the model.

Three SQLite features do real work here. WAL mode (PRAGMA journal_mode = WAL) means readers never block a writer, with the write-ahead log checkpointed back into the database files on stop. FTS5 gives every text-bearing entity a full-text shadow table, searched only through consent-checked queries — sealed (encrypted) columns are structurally banned from ever landing in one. And CHECK (json_valid(...)) constraints guard every embedded JSON column, so malformed JSON simply can't be written.

§05 · content addressing

Name the bytes by what they are.

Most storage names a file by where it sits — a path, a bucket key, a row id. Change the location and every reference breaks. Content addressing is the idea git made famous: name the bytes by what they are, using a hash of their contents. The name and the data become the same fact, so integrity checks and deduplication fall out for free — two copies of the same photo are just one blob.

Centraid identifies every blob by its sha256, giving it a blob:sha256-<hex> URI. On disk that lands under a two-character fan-out directory (so no folder holds too many files), written to a temp path and then renamed into place — an atomic move, never a partial write.

Two twists make this more than a git trick. First, Centraid owns the meaning and rents the bytes: a row can re-home its bytes to any S3-compatible remote — AWS, MinIO, R2, B2 — without touching a single relationship, and the gateway never trusts a remote's ETag; it re-hashes from its own local spool before believing anything matches. Second, the claim pattern: raw bytes arriving over the wire is not itself a vault write — the typed command that claims them is. Uploads stage under a staged_sha, and a single transaction promotes that stage into real content rows. Bytes are only ever servable once a reference — a content link, a folder tag, a cover — actually claims them; egress is derived, never just granted.

§06 · peer-to-peer

Two phones behind NAT, no middleman.

Your phone and your desktop almost certainly have no public address — both sit behind NAT, a router quietly hiding them from the open internet. That single fact is why personal software normally rents a cloud middleman just to let your own devices talk. Centraid instead rides iroh, an open-source peer-to-peer library, over QUIC — the modern transport that runs on UDP with TLS 1.3 built in and can carry many independent streams on one connection, so a live SSE feed is just a stream that stays open.

the fast path

Direct, punched

Both devices are behind NAT with no public IP, so iroh punches a direct hole through it — traffic goes device to device, no server in the middle.

the fallback

Relay fallback

When punching fails — symmetric NAT, strict firewalls — traffic relays through iroh's public n0 network, still end-to-end encrypted. Centraid rents no relay fleet of its own.

the identity

Keys are addresses

A device's network identity is its Ed25519 public key. The handshake proves it — no account, no TLS certificate, no exposed port.

Joining that network is a one-time pairing ticket, not a login: desktop shows a QR code that burns on first use, or a headless gateway mints a pasteable ticket over SSH that pins the gateway's key up front, closing the trust-on-first-use gap entirely. From there, enrollment is the whole access-control list — one revocable row per device key per vault, no roles, no permission matrix. Revocation bites mid-session, re-checked on every stream, and an unenrolled key is refused at the transport layer with a QUIC close code 401, before it ever reaches an application. The result is a star, not a mesh: Centraid refuses the hardest problem in personal software — multi-master sync — by never having it. One always-on gateway per person; every device is just a client around it.

§07 · oauth & pkce

Sign in at the source, not into the app.

An automation that wants your mail or your bank feed needs credentials — but handing it your password hands it everything, forever. There has to be a way to grant a narrower, revocable slice of access instead.

OAuth 2.0 is the standard answer: you sign in at the provider, not inside the requesting app, and the app receives a scoped, revocable token instead of your password. PKCE (RFC 7636, "Proof Key for Code Exchange") closes a gap for apps that can't hold a secret: the app generates a one-time code_verifier, sends its hashed S256 code_challenge up front, and proves at the end that the same app that started the sign-in is the one finishing it — so a stolen authorization code is useless to anyone else.

Centraid is BYO OAuth client in v0 — you register your own client with the provider; there's no Centraid-hosted relay standing between you and it, on purpose. The gateway runs the ceremony over a loopback callback with single-use state, then seals the resulting tokens at rest and refreshes them single-flight so two requests never race. The part worth teaching: automation code never touches a token — it writes a placeholder like {{connection:mail}}, and the broker injects the real credential outside the sandbox, refusing any host not on that credential's allowed_hosts pin. Unless a credential explicitly opts into writes, the injected fetch stays read-only.

§08 · sealed secrets

Encrypted at the column, not the disk.

Full-disk encryption protects a stolen laptop; it does nothing once the app is running and the disk is already unlocked. Passwords, card numbers, TOTP seeds — the columns that actually hurt if leaked — need protection that survives a compromised query, not just a compromised drive.

Centraid encrypts those columns cell-by-cell with AES-256-GCM, an authenticated cipher: tampering is detected, not just hidden. Each ciphertext binds its AAD to table.column:rowid, so a cell copied into another row simply fails to decrypt. Reads show the placeholder «sealed» by default; seeing the real value is its own consent verb, reveal, and every reveal is receipted. Sealed cells are structurally invisible to full-text search, and the audit journal stores keyed hashes of them, never plaintext.

The sealing key itself (the DEK) lives in a keys/ directory deliberately outside the vault — so the obvious backup move, copying the vault directory, carries ciphertext only. Key export, restore and rotation are shell-only: centraid-gateway key status|export|restore|rotate. The same boundary powers the Locker app's TOTP codes (RFC 6238) — the open standard behind every authenticator app, an HMAC over a 30-second time counter. Centraid computes the 6 digits inside the sealed boundary against the RFC's own test vectors; only the digits ever emerge, receipted, and redacted from the transcript.

§09 · consent lineage

"Ask permission" already had a body of standards.

Centraid didn't invent consent-as-infrastructure — it composes a handful of open standards for provenance, permission and disclosure that already answer pieces of the problem.

Standard What it contributes
W3C PROV Who touched what — entity, activity, agent, wasDerivedFrom. Centraid's provenance rows follow this shape, then hash-chain each row to its predecessor for tamper-evidence, beyond vanilla PROV.
W3C ODRL A machine-readable language for permission rules. Grant scopes and row filters are ODRL-shaped constraints, not free-text policy.
W3C DPV A shared vocabulary of purposes. Every grant is purpose-bound — dpv:Billing, dpv:HealthMonitoring — not just scope-bound.
Kantara Consent Receipts Every access decision leaves a receipt, chained to its provenance row.
Solid The pod model: a vault is a personal data boundary you own, and app grants echo Solid's app-grant shape.
GDPR Arts. 5, 20, 22 — storage-limitation policy rows, portable export jobs, and explanations for automated decisions.

Where Centraid diverges is enforcement: there's exactly one door, and a grant is evaluated on every read and every action — consent is data the data carries, not a setting checked once at login. Commands judged risky park for the owner to decide instead of running. And anything bound for the outside world — an email, a bank transfer — is staged as an inert outbox artifact holding placeholders, never live credentials, until the owner acts on it.

§10 · agents & mcp

Same three tools, whichever model is driving.

An LLM agent is just a model in a loop: read the situation, decide, call a tool, observe what came back, repeat. The open question every agent framework has to answer is how tools get handed to the model in the first place.

MCP (Model Context Protocol) is the answer that stuck — an open protocol for describing and calling tools that any model can use the same way, regardless of which lab built it.

Centraid's agent runs in-process against exactly three tools, served over an in-process MCP server: vault_sql (consent-checked reads), vault_invoke (typed commands) and vault_content (bytes). That surface is identical whether the model behind it is Claude, Codex, or the OpenClaw plugin. Every vault command is MCP-shaped — a JSON-Schema input/output contract plus declared pre/postconditions — but Centraid's twist is durability: a tool call isn't an ephemeral protocol message, it's a row. Every invocation is checked, receipted and auditable after the fact, and SSE streams both chat turns and automation runs live as they happen.

the composition

Ten borrowed ideas, one new thing.

None of these primers describe a Centraid invention. The contribution is the composition: one ontology drawn from the standards, in one SQLite file you can copy, with bytes named by hash, devices named by key, credentials brokered by the rules, secrets sealed cell-by-cell, consent checked at one door, and an agent that acts only through typed, receipted commands. Each idea is decades old or battle-tested; stacking them is what makes personal software ownable.
