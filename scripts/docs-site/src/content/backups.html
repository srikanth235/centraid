Chapter 4 · backups

One home for your data. One copy that isn't there.

The gateway is the sovereign control plane for your data — and one machine is still one failure domain. A dead disk, a stolen laptop, a bad rm don't ask permission. Offsite custody turns that loss into an outage: databases and code travel in encrypted snapshots, while attachments are born as sealed bytes in your provider's content-addressed store. There's one home for those bytes — on this device or hosted — and the provider sees storage, never content or keys.

§01 · why

A snapshot is everything, sealed.

A backup only counts if restoring it gives you back the thing you actually had — not a database with no apps, or a Locker full of ciphertext nobody can open. A Centraid snapshot takes that literally:

What's in it Source Why it has to be there
The vault vault.db + journal.db, staged point-in-time Every row, every receipt — the record itself.
Attachments local-primary objects, or only the bounded outbox not yet in a remote-primary CAS Photos, documents and receipts remain recoverable without duplicating remote bytes in every snapshot.
Installed apps a bundle of the gateway's app code store A restored vault with no apps installed is data with no way to see it.
The sealing key the vault's sealed-columns key, which normally lives outside the vault directory Without it, a restore brings back Locker's passwords as permanent ciphertext — a backup that looks complete and isn't. This is the entry that makes it a backup and not a placebo.

And what never leaves your machine unencrypted: everything. Chunking, encryption and the manifest all happen before a byte reaches the network — a provider stores opaque, content-addressed ciphertext, never plaintext, never a key. Full format: packages/backup/FORMAT.md.

§02 · setup

Turn it on. Pick where it goes.

Backups are a gateway config block, same discipline as everything else here — declared, not discovered:

gateway configbackup
backup: { enabled: true, keyringPath: "~/.centraid/backup-keyring.json", provider: { kind: "local", dir: "/Volumes/backup-drive" } // or: { kind: "remote", endpoint: "https://api.clawgnition.com", apiKey: "…" } } // Per-vault cadence and byte budgets live on the Backups page: // RPO · snapshot/verify cadence · cache/outbox headroom · bandwidth

The same shape covers both cases the config supports: a local-folder provider — an external drive or a mounted network share, the zero-setup default — and a remote provider speaking the open storage protocol, where endpoint and apiKey are all the gateway needs to know. Clawgnition is the first commercial one: client-side encryption (the same as the local case — Clawgnition never gets a key), S3-direct writes straight to Cloudflare R2, zero-egress restores.

There's no per-attachment routing to reason about any more. A vault is either On this device or Hosted — a single home connection, one managed provider bundle that carries snapshots, blobs and their derivatives together. You wire it in Settings → Account → Storage: under Hosted storage, Connect your storage provider (a name, a provider URL, an access key), then flip This vault from On this device to Hosted. Bring-your-own buckets and per-use assignment are gone — one home, or none.

backup CLI · admin shell
$ centraid-gateway backup status --data-dir ~/centraid-data --vault Local
$ centraid-gateway backup run --data-dir ~/centraid-data --vault Local
$ centraid-gateway backup list --data-dir ~/centraid-data --vault Local
$ centraid-gateway backup restore-verify --data-dir ~/centraid-data --vault Local
# status · run · list · verify · restore · restore-verify · kit — one verb per concern
# blank machine? centraid-gateway recover --kit … --api-key … --data-dir … (top-level verb)

The keyring (keyringPath) is the one file in this whole setup that isn't just convenience: it holds the key material that makes every snapshot readable. It's covered on its own, next.

§03 · the recovery kit

Export it now. Not when you need it.

The keyring is a small JSON file holding your key material as a set of epochs — the active encryption key plus every prior one still needed to read old snapshots. Rotating the key (say, after a suspected compromise) mints a new epoch rather than discarding the old one, so nothing already backed up becomes unreadable.

centraid-gateway backup kit exports a recovery kit: the full keyring plus enough provider metadata (endpoint, target id, a label — never the provider api-key, which is rotatable server-side and re-issued by logging in) to restore from nothing but this one file on a blank machine. That's the acceptance test for the whole feature: kit in hand, no other state, full restore.

Export it the day you turn backups on, store it offline — printed, a hardware key, a safety-deposit box, anywhere that isn't the disk it's protecting — and treat it exactly like the credential it is. Losing every key epoch means losing the backups. By design. A provider that could recover your key for you is a provider that can read your data.

§04 · restore

Never onto the live vault. Never a silent resend.

centraid-gateway backup restore materializes a snapshot into a fresh directory — it never touches, never overwrites, never swaps the running vault. It's lazy by default: rows and code come down first, and every blob the provider's attested inventory already holds is fetched on first access rather than pulled up front — --full forces eager materialization, --at <time> restores to any moment between snapshots (PITR). Before it downloads a single chunk it gates on compatibility: an unknown format, or a snapshot written by a newer ontology or vault schema than the running gateway understands, is refused outright. This is v0 — there is no migration path, only "update the gateway first."

Restore is the surgical tool; recover is the front door. A truly blank machine — new disk, nothing but a recovery kit and a provider access key — runs one top-level verb, centraid-gateway recover, and the desktop wraps the very same service in a first-run gate: Start fresh, or Recover my vault. Drop the kit, enter your Access key, Find my vault, then Recover this vault — and if the provider prices egress, a metered-recovery confirm stands between you and the first billed byte. What you watch is three honest stages — Fetching your vault → Replaying recent changes → Warming previews — before Enter Centraid. Underneath, the engine walks discovering → fetching → replaying → fencing → adopting → warming: it settles compatibility from registry metadata alone (a refusal never bills a metered home), replays the WAL to the tip, and — lazy by default — leaves the bytes the attested inventory holds to arrive on demand. The vault is live before all of it is local.

1

The quarantine

A restored vault comes back holding yesterday's world — including an outbox full of drafts and automations ready to fire the instant they mount. Restoring a backup must never re-send an email, so the engine marks the restore and the gateway, on first mount, parks every outbox row, disables every automation, and flags every connection for re-auth review. You re-arm each one deliberately, not by accident of timing.

2

Takeover, loudly

Restoring onto a new machine and resuming backups from there is a takeover. The recover engine fences it: it reads the provider's current generation counter and seeds its own state at generation + 1, so the recovered machine owns the next snapshot. If the old machine is still running and tries to back up, its registration 409s — and it's required to surface "another machine has taken over this vault" and stop, never silently retry with a bumped number. Split-brain becomes a notification, not a silent fork.

3

Orphan-grace, so PITR stays honest

Lazy restore means bytes keep trickling in after the vault is already live, so the custody sweep can't treat a not-yet-fetched blob as garbage. Orphan-grace GC holds orphaned blobs for the full recovery-window duration — the same span the provider lets you restore across — so a point-in-time restore between two snapshots never finds its bytes reaped out from under it.

§05 · health

Cadence, custody and proof. One screen.

Custody has its own screen now — Operations → Backups, headed "Backups" with a Protected by {provider} badge and a Manage link straight to Settings → Storage. One derivation renders five plain-language metrics: Freshness (how recently you were provably protected offsite — the tightest of four custody clocks), Recovery window (how far back you can restore), Privacy (a structural constant: sealed bytes, client-only keys), Cost (bytes stored against quota), and Exit (Export everything, always, with restore egress priced honestly). An inline gate keeps the recovery kit honest too — Export recovery kit / I've saved my recovery kit.

A collapsed Diagnostics panel holds the levers and the raw clocks: Verify now, Back up now, and a four-cell grid — Newest snapshot, Last verification, Newest WAL segment, Outbox drained — plus per-vault policy and what the provider holds: provider-attested (or raw-bucket computed) object and byte totals, recent snapshots, the WAL/PITR span, prune history, and any missing or orphaned objects. A weekly reconciliation runs before eviction can trust remote custody; if a provider silently loses an attachment, the gateway demotes that replica immediately and pins any remaining local copy — it never auto-deletes an orphan. “Confirmed when offsite” is a completion policy, not a long-held request: a provider outage never rolls back a durable local claim.

Verification is the one the protocol insists on, because the provider contract is explicit that providers verify nothing on your behalf — no chunk-existence check at commit, no manifest parsing, no server-side integrity sweep. That is the deliberate division of labor: the provider owns durability and lifecycle bookkeeping, the client owns knowing whether the bytes are actually still good. On verifyEveryDays, the gateway HEADs every object the newest manifest references and downloads + decrypts a random sample, recomputing their keyed ids. A backup you've never verified is a backup you're only assuming works.

§06 · provider ecosystem

The seam is open, on purpose.

Backups follow the same anti-lock-in instinct as connections and connectors: the wire contract between a gateway and a provider — centraid-storage-provider/1 — is a published, versioned protocol, not a Clawgnition implementation detail. Any operator fronting S3-compatible storage can implement it: discovery and capabilities, two auth tiers, target lifecycle, generation fencing, the GC min-age rule. A reference conformance kit runs the same assertions against any implementation; passing it is what "certified provider" means.

What differs between providers is declared, not guessed: retention (how aggressively older snapshots get pruned into a ladder — keep-all, then daily, then weekly) and restore cost class (free-egress vs. metered) are both part of the discovery document a client reads before you commit to a plan, and both are things the gateway shows you — not fine print you find out about during a 300 GB restore.

Underneath, the provider fronts S3-compatible storage on your behalf — the gateway never handles raw bucket credentials, and there is no bring-your-own-bucket connection to wire up. That's what the home connection buys: one managed bundle covering every store class (snapshots, CAS, derivatives), reached over the wire routes under /v1/storage/*, with the gateway holding only an endpoint and an access key.

Normative text lives in the protocol source, not here: packages/backup/PROTOCOL.md (the provider seam) and packages/backup/FORMAT.md (the snapshot format itself). This chapter is the tour; those are the spec.

§07 · honest caveats

What backups don't smooth over.

A few things worth knowing before they surprise you:

irreversible

Purge means purge

Deleting a target is soft and undoable within a declared window. Purge is not — it's a one-way erasure that permanently blocks undelete, and the protocol requires it to run through the provider's own interactive session, never an api-key call. A compromised gateway credential cannot destroy your backup history; only you, logged in directly, can.

retention

History gets pruned

A provider's retention ladder means old snapshots don't live forever — the newest is always protected, but the daily-then-weekly thinning is real. If you need a specific point in time kept indefinitely, that's a provider policy question, not a gateway setting.

first upload

Seeding still takes time

The first backup of a vault with hundreds of gigabytes of photos and documents is a full upload — hours, not minutes, depending on your connection. A capable paired device hashes, seals and uploads attachments directly through presigned URLs (including native iOS/Android background transfer), so the gateway's disk and uplink are not the bulk-byte path. Thin clients use the bounded gateway outbox; when that honest transit budget is full and the remote is down, admission waits instead of risking ENOSPC or loss.
