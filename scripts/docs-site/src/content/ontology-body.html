§01 · Thesis & system map

One life. One model. Apps become projections.

Your life is currently modeled by whoever shipped the app: your heart rate belongs to a watch vendor, your invoices to an accounting SaaS, your relationships to a contacts silo. Centraid inverts the ownership. A single, person-owned, self-hostable ontology holds the canonical record — identity, place, time, money, content, activity, observation, relationship, classification — and every familiar app becomes a thin projection over it. A personal agent operates across the whole model through one typed, consent-governed surface.

The product is the substrate, not the apps. Familiar interfaces lower adoption cost; one shared model removes the silos; the consent and agent planes make it sovereign and agent-ready. Read it top-down as a user, bottom-up as an architect: surfaces render projections; projections read extensions; extensions never fork identity — they hold a foreign key into the spine. Nothing crosses a layer without passing the consent plane, and agents only touch the model through the command surface.

§03 · Detailed schema

The logical model, generated from data

Physical target — local-first SQLite

STRICT tables throughout; primary keys are TEXT UUIDv7 (time-ordered, offline-mintable); money is never floating point — fixed-scale integers (amount_minor INTEGER + ISO 4217 currency); all timestamps are TEXT ISO-8601 UTC with named IANA zones stored separately; structured columns use json1 with CHECK(json_valid(…)). Convention (elided per-table for brevity): every table additionally carries tenant_id TEXT NOT NULL REFERENCES core.vault when hosted multi-tenant — locally, the file itself is the vault. Append-only by contract: consent.receipt, consent.provenance, agent.invocation_check, agent.evidence and core.observation amendments never update in place; mutable projections keep ordinary created_at / updated_at. SQLite has no row-level security, so consent is enforced in the gateway: every query and every typed command is evaluated against consent.grant_scope before it touches a connection, and every allow/deny emits a consent.receipt. Bytes behind content_uri replicate via ordinary blob tooling — they are rented; the rows here are owned.

Physical layout — two files, split deferred. All eleven schemas live in one vault.db: every command commits provenance, core rows and domain rows in a single ACID transaction, and all 133 foreign keys stay engine-enforced. The append-only audit stream — consent.receipt, consent.provenance, agent.command_invocation / invocation_check / evidence / explanation — lives in a separate journal.db: it grows orders of magnitude faster than the model, never rolls back domain writes, and keeping it out of the vault keeps the sovereign asset permanently small. Export = copy two files and verify hashes. Because every table is schema-qualified and only the gateway holds connections, a later per-domain split (one file per schema via ATTACH) is a mechanical migration held in reserve — justified only by a concrete trigger such as a separate encryption key for health, media metadata dominating backup time, or multi-device sync conflict reduction. Generated apps that need real tables get a per-app appext_<app_id>.db attached alongside: its tables may hold FKs into the vault, the vault never references them, and uninstalling deletes the file with zero residue.

Since v1.1 — what the implementation added, and two standing stances

New canonical surface. core.collection (one curation mechanism, #274) and standoff link anchors (#282); the sync spine — sync.connection, the external-id map, staged import batches — with broker-owned credentials as sealed sidecars (#290, #304); the sealed column class (AES-GCM at rest, explicit reveal verb, structural FTS bar, #293); blob custody with content derivatives (#296); the enrichment spine (derived rows land in tables the ontology already has, attributed and re-derivable, #299) now under per-class standing consent (#310); the outbox — external writes as inert artifacts the owner decides on, standing (actor, verb, target) grants, typed graph refs and drain-side publish-back (#306, #308, #310); consent memory (scope tombstones + widening requests, #308); and three new domains — People, Locker, Tally — each re-anchored by #310: settlements post to the canonical ledger, groups decorate social.circle, locker items anchor their broker connection and tag through core.tag.

Stance — the runtime store stays outside the graph. Conversation transcripts and automation run summaries live in the conversation-ledger band of the per-vault journal.db (formerly a standalone transcripts.db), deliberately not in the model (issue #280: the vault directory is the sovereign unit; the runtime store is operational state). They are activity-shaped, so the pull to join them will grow — the decision of record is: they stay out until an agent query genuinely needs them, and the door in is a thin core.activity projection, never soft TEXT joins from canonical rows.

Stance — version brokering waits for v1. Rows and commands stamp ontology_version (now 1.3), but the gateway enforces EQUALITY, not R07's compatibility windows: pre-release there is no data to migrate and no third-party contract to keep rendering, so a mismatch means a stale registration. The windows return when v1 ships; until then the ladder itself is one rung — schema modules carry their final shape and a dev vault is recreated, not migrated.

The gateway contract — everything SQLite cannot enforce, in one auditable choke point (elaborated in §10)

SQLite has no row-level security, no stored procedures and no server, so the gateway is the sole holder of connections — locally and hosted — and every request passes one pipeline: identity → consent → contract → execution → evidence. Identity: every caller authenticates as a row (consent.app.signing_key, agent.agent, consent.device); no credential, no path to data. Consent: an active access_grant must exist, its grant_scope rows must cover the schema and verb, row filters and field masks are applied, declared purpose is checked against policy, and command risk against the app’s risk_ceiling — any step may deny, and a deny is a receipted outcome, not an exception. Contract: writes are never rows, only typed commands — payloads validated against JSON-Schema, preconditions evaluated as real queries and recorded as agent.invocation_check rows before anything mutates, postconditions verified after. Execution: the gateway owns write ordering (journal invocation → vault rows → domain rows), the ACID boundary, idempotent replay off UUIDv7 keys and external_id uniqueness, and everything declarative constraints cannot express — polymorphic refs (link, attachment), cross-file FKs, Σ-invariants. Evidence: every read and every command, allowed or denied, emits a consent.receipt; every write stamps consent.provenance; outcomes hang agent.evidence and agent.explanation — unskippable because there is no other door.

Standing duties (not request-shaped): compiling and clamping consent.app_view definitions and refreshing materializations marked wasDerivedFrom; parking high-risk invocations and routing owner confirmations (the pause between draft and send is gateway state); the revocation cascade — sessions killed, views invalidated, extension files detached, instantly; lifecycle sweeps (purge_at deletions, retention policy, share and grant expiry, export_job runs); ingest discipline at the vault border (dedupe on external_id, per-row provenance, handle→identity resolution); ontology version brokering per app (R07 compatibility windows); and file custody — attachments, WAL checkpoints, backups, appext creation and deletion.

What the gateway must not do: no domain logic (state machines and command implementations are domain-owned; the gateway hosts and checks them), no reasoning (planning and judgment belong to the agent; agent.judgment rules are consulted as constraints, never authored), no rendering (projections are app-side), no byte custody (content_uri targets are brokered, not stored). The trade is explicit: engine enforcement is exchanged for a single testable choke point — so a gateway bug is a sovereignty bug, which is exactly why it stays a thin, mostly declarative interpreter over the consent and capability tables, with the model, not the code, as the source of truth about what is allowed.

§04 · Modeling principles

Five commitments the model never breaks

Everything in §03 is a consequence of these five decisions.

P1

Shared where meaning is stable, local where behavior differs

“A person,” “a place,” “money moved,” “something was measured” — these mean the same thing in every app, so they live once, in the core, carrying identity and history. What differs per domain is behavior: a workout’s training load, an invoice’s dunning rules. Domains add thin typed extensions — a workout holds a FK to the activity it is — and never fork identity into a parallel copy.

P2

Own the meaning, rent the bytes

The durable asset is the graph: rows, relationships, provenance, classifications. Pixels, PDFs and audio sit behind content_uri with a sha256 integrity hash — storable on any disk, NAS or object store, and re-homeable without touching a single relationship. Losing a blob store loses bytes you can re-fetch; losing the model would lose the meaning. So the model is what you own.

P3

Consent travels with the data

Sovereignty is a plane, not a settings page. Apps hold scoped, expiring grants (purpose-bound per W3C DPV, expressed as ODRL-style rules); shares, retention and residency are policy rows; every access decision leaves a Kantara-style receipt chained to W3C PROV provenance. A grant is evaluated on every read and action — consent is data the data carries.

P4

Agents act through commands, never tables

The agent plane exposes one uniform surface — discover, query, reason, act, verify, explain, learn — where every action is a typed command (an MCP tool) with declared pre- and postconditions. Agents never hold a write handle to a table: they invoke agent.commands that are consent-checked, executed transactionally, provenance-stamped and receipted. Corrections feed back as durable judgment rows, not vibes.

P5

Entity-scoped meaning joins, surface-scoped state rides

An owner judgment about an entity — starred, a memo, a relationship — means the same thing in every app that renders the entity, so it lives in the universal joins (core.tag, knowledge.annotation, core.link), keyed to the canonical row. State that only means something inside one surface — pinned-to-top, sort order, a cover image — rides the domain’s own rows. A flag never mints a column, a row or a table: Gmail stars ≠ Drive stars ≠ Photos favorites is a fragmentation artifact of separate databases, not a design to copy. Reference products are evidence, not authority.

§10 · The gateway

One door. Every request. No exceptions.

SQLite has no row-level security, no stored procedures and no server — so everything an enterprise database would enforce about who may do what lives in exactly one place. The gateway is the sole holder of connections, locally and hosted: apps, agents, workflows and generated views never see a file handle. Every request — human tap, agent plan, cron trigger — walks the same five-stage pipeline. Click a stage to inspect what it checks, which tables it consults, and what a denial looks like.

Standing duties — the work between requests

Not everything the gateway does is request-shaped. Eight duties run on schedules, triggers, or lifecycle events — each one still writes receipts and provenance like any caller would.

View service

Compiles consent.app_view definitions, proves every join follows a declared FK, clamps results to the owning app's scopes, refreshes materializations, and marks cached rows wasDerivedFrom so they stay purgeable.

Confirmation routing

Parks any invocation above the caller's risk_ceiling, surfaces the confirmation to the owner, then resumes or expires it. The pause between draft and send is gateway state.

Revocation cascade

Revoking a grant is instant and total: sessions killed, materialized views invalidated, appext files detached. The §11 success test — projections go dark, model and receipts remain — is this cascade.

Lifecycle sweeps

Executes purge_at deletions, enforces retention and residency policy, lapses shares and grants at expires_at, and runs export_jobs — assemble files, verify hashes, mark the vault exported.

Ingest customs

The border post: import batches dedupe on external_id, stamp per-row provenance, and resolve raw handles to identities via party_identifier — OFX, ICS, vCard and IMAP all enter the same way.

Export & portability

Owns the lossless round-trip: any scope of the model and its bytes out as verifiable files, and back in with identities intact — the test that gates every new domain.

Version brokering

The ontology versions explicitly (R07); the gateway negotiates which contract version each app speaks and serves compatibility windows during migrations, so old projections keep rendering.

File custody

Which SQLite files exist, what is attached when, WAL checkpointing, backup coordination, and per-app appext_<app_id>.db creation and deletion.

The negative space — what the gateway must never become

The trade is explicit: engine enforcement exchanged for a single testable choke point. A gateway bug is a sovereignty bug — which is exactly why it stays a thin, mostly declarative interpreter over the consent and capability tables. Four things are permanently out of scope:

No domain logic — state machines and command implementations are domain-owned; the gateway hosts and checks them No reasoning — planning belongs to the agent; agent.judgment rules are consulted as constraints, never authored No rendering — projections are app-side; the gateway serves rows and outcomes, never pixels No byte custody — content_uri targets are brokered, not stored; meaning is owned here, bytes are rented elsewhere

§11 · Implementation rules

Ten hard rules, then a first boundary

The stance in §01 survives contact with engineering only if these are enforced in code review, not aspiration.

IDs are immutable and meaningless

UUIDv7 TEXT keys, minted once, never recycled, carrying no semantics. Merges keep both rows and record a core.link of relation same-as with provenance — history is never rewritten.

Extend, don’t fork

A domain adds columns by holding a UNIQUE NOT NULL FK to the core row it specializes (health.workout.activity_id). Creating a second “person”, “event” or “transaction” table is a rejected review.

Relationships are typed and temporal

All cross-entity meaning goes through core.link: a SKOS-governed relation concept, valid_from/valid_to, who asserted it, and provenance. No boolean junction tables that forget when and why.

Writes are typed commands

Apps and agents mutate state only via registered agent.commands with JSON-Schema inputs, declared pre/postconditions and idempotency class. Raw UPDATE is for migrations, run by the owner, and it too leaves provenance.

Durable facts vs rebuildable projections

Observations, transactions, messages, provenance: durable, append-mostly, exported. Feeds, dashboards, search indexes, “smart albums”: projections that can be deleted and regenerated. Never let a projection become the only home of a fact.

Consent travels with the data

Every read path and every command execution evaluates consent.grant_scope (purpose, verbs, row filter, field mask, expiry) in the gateway and emits a consent.receipt. A share or export carries its policy with it.

Version the ontology explicitly

Every row stamps ontology_version; migrations are forward-only scripts registered per version. An agent must refuse a command whose schema version it doesn’t recognize rather than guess.

Corrections become durable judgment

When the owner fixes the agent — recategorizes a transaction, un-tags a face — the diff is stored as agent.correction and distilled into a scoped agent.judgment rule the agent consults before acting again. Learning is rows, auditable and revocable.

One judgment, one mechanism

Entity-scoped meaning goes through a universal join; surface-scoped state rides the domain row. Starred is a flags-scheme core.tag on the canonical entity; an owner memo is a knowledge.annotation; employment is a core.link. A favorite boolean on a domain table is a rejected review — it silently discards who judged and when, and re-imports the app-silo fragmentation the vault exists to dissolve.

Generated apps ride the paved road

Vibecoded apps are untrusted code by construction, so they get no special path: a manifest becomes a grant (origin=generated, explicit risk_ceiling), reads go through registered consent.app_view definitions the gateway clamps to scope, writes go through existing typed commands, and custom data lives in core.attribute/concept/link or a declared per-app ext band inside the vault (issue #286) that may point into the canonical model but is never referenced by it. Uninstall = revoke grant + retain-then-purge the band; when apps are cheap to generate, projections become disposable and the model becomes everything.

The domain admission checklist (issue #310 C6)

Locker and Tally shipped whole apps before anyone noticed the universal mechanisms had not come along — a second tag table, a private junction, an unindexed domain. New domains keep landing; drift is a process failure, not a modeling one. A PR that adds a vault domain answers all seven in review:

1 anchor — does every entity extend or reference a core row (R02), or carry a written exemption?  2 relations — do cross-entity joins ride core.link / an existing mechanism, not a new junction (R03)?  3 flags — stars and booleans through core.tag flags (R09)?  4 memos — owner remarks through knowledge.annotation?  5 search — an FTS spec for its text (sealed columns structurally barred)?  6 registry — entities in tables.ts, commands registered, scopes declared?  7 doc — this document updated in the same PR?

Recommended first boundary

Ship the spine where the pain is cheapest and the loop is fastest: core.party + core.event + the schedule domain, wrapped in the full consent plane. Contacts and calendar exercise identity, time, typed links (attendee), one real state machine (RSVP), imports with provenance (vCard, RFC 5545 ICS), and two honest agent commands — schedule.propose_event and schedule.reschedule_event — each consent-checked and receipted end to end.

Success test: revoke the calendar app’s grant and watch every projection go dark while the model, history and receipts remain yours. Then add the second domain — and do not add a third until export→reimport round-trips losslessly. Once the round-trip holds, expand to the foundation release: schedule, social, finance and health together, because those four generate the densest cross-app context an agent can act on.

§12 · Inside the runtime

A second plane, not a second silo

Centraid’s app host (@centraid/gateway + @centraid/app-engine) once gave every app a private data.sqlite — full SQL, zero sharing. Issue #286 deleted that silo: app-declared tables now live as a registered ext band inside vault.db (ext.<app_id>.*), diffed on publish, retained-and-purged on uninstall. A handler reaches everything through ctx.vault — canonical entities under consent scopes, its own ext tables under its band — and every call walks §10’s five-stage pipeline on the host side of the process boundary.

The seam — a second RPC channel beside db

Centraid handlers run in a worker thread that holds no file handle: every db call round-trips as a message to the host, which owns the SQLite connection (app-engine/src/handlers/handler-runner.ts host-side, app-engine/src/worker/runner.ts worker-side). ctx.vault is the same move aimed at the canon: the worker posts {type:'vault'} messages with read / invoke / query / describe payloads; the host resolves the running app to its Credential, calls the process-held vault gateway (vault/src/gateway/gateway.ts), and replies with allowed rows or a receipted denial. Consent is enforced on the host side of the boundary — the correct side, because the worker gives crash isolation and a controlled surface, not a security sandbox.

No keys in the worker — credentials resolve host-side; the handler surface carries capability, never secret No second wire — ctx.vault rides the existing worker RPC; no new protocol, no new port No FK across bands — a declared ext.* table may point into the canon; the canon never points back (R09, band-shaped since #286) No unreceipted path — every ctx.vault call ends in a consent.receipt, allow or deny No consent shortcut — a generated app’s call is a §10 pipeline walk like any other caller’s

Who is who — the concept mapping

Each vault-plane concept lands on a Centraid structure that already exists. Nothing here invents a parallel mechanism; the integration is mostly wiring.

consent.app ← registry entry

An app that goes live (registry.ensureUploaded) is enrolled via enrollApp as a consent.app row (origin=generated for vibecoded apps). App deletion runs the revocation cascade: views dark, parked invocations dropped, extension file deleted.

signing key ← identityDb

The minted signingKey persists host-side in the gateway’s identity store, keyed by app id. It never crosses into the worker; the host attaches it when proxying a ctx.vault call.

agent.agent ← chat & automation runners

The conversation runner and automation fires act as an enrolled agent.agent riding an owner device (session binding). Risk ceiling medium; high parks for owner confirmation — the send-message pause, verbatim.

owner device ← the host itself

bootstrapVault mints the first device on first boot, idempotently. The desktop user’s direct actions — confirmations, grants, revocations — ride it with riskCeiling='owner'.

vault.db + journal.db ← GatewayPaths.vaultDir

A new caller-derived path slot beside appsDir, identityDb and analyticsDb. Opened by buildGateway(), WAL mode, closed on stop(). The vault stays a file the owner can copy out.

agent.command ← app.json action

Both are JSON-Schema-validated typed writes. A declared action may delegate to a vault command; the vault then adds consent, pre/postconditions, provenance and receipts on top of Ajv input validation.

ext band ← declared ext.* tables

R10’s per-app extension surface is no longer a separate file: issue #286 moved app tables INTO vault.db as a declared, registered band. Ext tables may FK the canon (and each other within one app); the canon never references them; uninstall = revoke grant + retain-then-purge the band.

standing duties ← in-process scheduler

Centraid’s gateway already runs a minute-boundary cron scheduler. sweep() joins its tick; checkpoint()/backup() join the lifecycle; revocation is wired to app deletion.

Six phases, each independently shippable

Sequenced so the risky parts — the handler primitive and the consent UX — land only after the vault is simply present and app identities exist. Every phase carries its own tests.

Host holds one vault

Add vaultDir to GatewayPaths; buildGateway() opens the pair, bootstraps the owner idempotently, registers the four foundation domains, exposes the vault gateway on BuiltGateway, closes it on stop(). Tests: builds with a vault, bootstraps once, survives restart.

Apps become consent.app rows

App-live enrolls the app and persists its signing key host-side; deregistration calls revokeGrant and the cascade follows. Tests: registration mints a credential; deletion goes dark.

ctx.vault in the handler surface

Worker proxy mirroring db: read / invoke / query / describe. The host resolves app → Credential and calls Gateway.read/invoke; a deny returns a shaped handler error carrying the receipt id. Tests: a handler proposes an event into the canon; an ungranted read is a receipted deny.

Grants, owner-approved

app.json declares requested scopes; the owner approves and the host calls createGrant. Deny-by-default until then — installation is not consent. Tests: pre-grant denied, post-grant allowed, owner reads unaffected.

Manifest & builder surface

A vault block in app.json (requested scopes + command delegation) and handler codegen that emits ctx.vault.* — declared handlers stay the agent’s primary surface; the builder generates the shape.

Duties on the gateway clock

sweep() joins the scheduler tick; checkpoint/backup join start/stop lifecycle; parked confirmations surface through the existing confirmation UI instead of a bespoke one.

Decided and built

Consent model: manifest-declared scopes with explicit owner approval. An app's app.json vault block is a request, never a grant; the owner approves it over POST /centraid/_vault/apps/<id>/grants, and until then every call is a receipted deny.

Data planes: one file, two bands. The per-app data.sqlite is deleted (issue #286); app extension tables ride inside vault.db as the declared ext band, and the shipped blueprints declare zero of them — pure projections. The canon stays the consent-gated shared plane; the band boundary, not a separate file, is the silo line now.

All six phases are implemented and tested: the vault plane mounts off GatewayPaths.vaultDir (gateway/src/serve/vault-plane.ts), live apps enroll and uninstalls cascade, ctx.vault rides the worker RPC (app-engine/src/worker/runner.ts), owner routes serve grants and parked confirmations, the builder prompt teaches generated handlers the surface, and the sweep clock runs on the gateway lifecycle.

And the §01 projection band ships as blueprints. Eight product surfaces are first-class templates in packages/blueprints/apps — docs, photos, notes, people, locker, tally, agenda, tasks — each a pure projection: zero private tables, every row from ctx.vault, every mutation a typed command, consent denial and the parked pause rendered as first-class UI states. Revoke a grant and the projection goes dark; the model, history and receipts remain. The same eight are documented from the user's side in the Apps chapter.
