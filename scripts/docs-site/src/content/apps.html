Chapter 2 · apps

Apps are projections, not places.

In the silo world, an app is where your data lives. Here it is a window: a folder of plain HTML and JavaScript that renders vault rows and mutates them through typed commands, under a grant you signed. Delete the app and nothing of yours is lost. Ask the builder to reshape it and it's a different window onto the same record by dinner.

§01 · the model

No private database. On purpose.

A Centraid app has no database of its own. The per-app data.sqlite of earlier designs is deliberately dead: every row an app shows comes from the vault through its granted scopes, and every mutation is a typed vault command. Handlers hold no file handle — they reach the canon through ctx.vault, and consent is enforced on the host side of that boundary.

When an app genuinely needs its own tables — a game's score sheet, a niche tracker — it declares an ext band: registered tables that live inside vault.db under the app's namespace. Ext tables may point into the canonical model; the canon never points back. Uninstall retains, then purges the band — the silo line is a band boundary you can see, not a separate file you forget.

§02 · the blueprints

Eight windows, one record.

The shipped apps are pure projections — and each one exists to prove a different part of the substrate. Star something in Docs and Photos knows, because a star is a canonical tag, not a column.

files that matter

Docs

A drive for important documents. Bytes dedupe by hash into canonical content; folders are concepts, not directories; trash keeps its folder for restore; PDFs preview in place. Proves: content custody.

the library

Photos

Albums are canonical collections that can hold mixed types; favorites live on the asset; face-region proposals wait for your confirm before touching a person. Proves: derivatives & enrichment consent.

thinking surface

Notes

Notebooks over canonical note rows; type @ to link any entity inline — the reference survives edits via standoff anchors, and orphaned links land in a strip instead of vanishing. Proves: universal cross-referencing.

personal crm

People

Every person is the same canonical party your calendar and expenses reference. Circles, cadences ("reconnect every 6 weeks"), important dates, gift ideas, debts, a journal. Proves: shared identity.

password manager

Locker

Logins, cards, Wi-Fi, notes — secrets sealed at rest, revealed per-item with a receipt, TOTP computed client-side, Watchtower flags weak and reused passwords without unsealing anything into a transcript. Proves: sealed columns.

expense splitter

Tally

Groups decorate canonical circles; expenses validate their splits server-side; settlements post to the real ledger and balances are derived, never stored. Proves: money as canonical transactions.

calendar

Agenda

Events, proposals, RSVPs on the canonical schedule; cancelling someone else's plans is exactly the kind of act that parks for your confirmation. Proves: confirm-gated commands.

task manager

Tasks

A Things-style board over canonical tasks — subtask families, priorities, a closed logbook — the same rows your assistant queries and your automations nag you about. Proves: one task list, every surface.

§03 · anatomy

Small enough to read in an evening.

An app is a folder with no build step: the runtime loads the files as they are. Four things matter —

apps/docsthe whole app
docs/
├── app.json       manifest: identity, scopes, actions, queries, knobs
├── index.html    the UI shell
├── app.js · app.css  vanilla JS controller + styles
├── actions/      one handler per intent — upload.js, rename.js…
├── queries/      bounded reads — drive.js, search.js
└── kit.js        shared UX kit, served canonically to every app

The manifest is the contract: a vault block declaring purpose and scopes (which tables, which verbs — read, act, reveal), actions with JSON-Schema inputs and an optional confirmation gate, queries that return bounded windows (never whole-table pulls), and knobs — font, width, color, view — that make customization a setting instead of a fork.

Handlers are pure functions: (ctx, input) → outcome. They run in a worker that holds no keys and no connections; every vault touch crosses to the host and comes back as executed, denied, or parked — with a receipt id either way. The kit gives every app the same toasts, confirms, charts, reference chips and Ask panel, so eight apps feel like one product.

§04 · the builder

Describe the change. Poke the preview. Publish.

Blueprints are starting points, not products you're stuck with. The builder is an agent that edits app code in conversation — "make Docs group by year", "add a warranty-expiry view to home inventory" — against a live preview, on a draft.

builder · docsdraft
you I want receipts separated from documents, and a monthly total next to each folder.
builder ⚙ vault_sql → folders, attachment roles, txn joins Receipts already carry a canonical role — no new table needed. Drafting a filtered view + a query that joins attachments to transactions. Preview updated. Publish?

The guardrails are the same ones everything else obeys: the builder reads the live schema and ontology vocabulary each turn, is steered to map onto canonical mechanisms first (an ext table is a justified escape, not a default), can't widen the app's scopes without a request you approve, and publishes through the app lifecycle — which re-diffs the grant. Generated code is untrusted by construction; it gets the paved road, not a side door.

§05 · attach & link

Bytes attach. Entities link. Everywhere the same.

Two verbs cover every "this belongs with that" in the system. Attach binds bytes to an entity: stage a file (up to 512 MB, deduped by hash), then claim it with a role — the receipt on a transaction, the manual on an appliance, the photo on a person. Link binds entities to each other through the canonical link table: typed, temporal, provenance-stamped.

Inline references are links with an anchor: type @ in Notes and the mention pins to a text span that survives editing. Every app renders references through the same resolver — a card that knows whether its target is live, trashed, missing, or simply not readable under your grant. A link you're not allowed to see through says so, instead of leaking a title.

Attach = core.attachment (bytes, role) · link = core.link (typed relation) · anchor = standoff selector. One judgment, one mechanism — rule 9 →

§06 · agents drive apps

Saying it and tapping it are the same code path.

An app's declared actions and queries aren't just its button wiring — they are its agent surface. A tap posts to the dispatcher; an agent's vault_invoke lands on the same dispatcher, the same schema validation, the same handler, the same outcome shape. There is no shadow API for machines, which means talking to an app is full-fidelity use — and every guardrail you've read about (scopes, parking, receipts) covers both doors, because there is only one door.

Inside each app, the kit's Ask panel runs a scoped conversation with that app's surface; parked writes come back as proposal cards you approve or discard in place. The vault-wide version of the same loop is the assistant — Data § the assistant.

§07 · on the phone

Written once, worn twice.

Apps are responsive by rule, mobile layout first: the same HTML that fills a desktop window reflows to the phone, and desktop-only flourishes gate themselves on width instead of forking the code. The mobile companion renders the identical app over the peer-to-peer tunnel — same handlers, same grants, live updates over SSE — plus an approvals surface, because a parked command should be answerable from your pocket.
