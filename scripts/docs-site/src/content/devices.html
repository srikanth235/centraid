Chapter 3 · devices

One gateway. Everything else is a client.

Centraid refuses the hardest problem in personal software — multi-master sync — by never having it. There is one always-on gateway per person, holding the vaults and running the automations; every device is a client in a star around it, connected peer-to-peer with no relay service to subscribe to. This chapter is that star, wire by wire.

§01 · topology

A star, never a mesh.

The gateway is the sole holder of vault connections — apps, agents, phones and laptops never touch a database file. That single fact buys the consent story (one door to check), the audit story (one place receipts mint), and the sync story (there is nothing to reconcile, because there is one writer). The same gateway core runs in two homes:

home 1

Inside the desktop shell

Embedded in the Electron app's main process. Zero setup — this is where everyone starts. Automations fire while the app is open.

home 2

A headless daemon

centraid-gateway serve on a spare machine or home server. Token persists, scheduler always ticking — the long-term home.

The trade is explicit: the gateway becomes the thing to protect — which is why backup is two files plus a key, and why admin actions never ride HTTP.

§02 · addressing

Clients speak to a (gateway, vault) pair.

A gateway mounts many vaults, so no request is ever "to the gateway" — it names its vault (x-centraid-vault) or is resolved by device identity, and the whole request runs inside that vault's scope. There is no server-global "active vault": two clients on two vaults can't cross-pollinate by construction.

who may enter

Enrollment is per-vault

A device's ed25519 endpoint id is enrolled against specific vaults in an ACL the admin CLI and the daemon both watch. Multi-vault access is multiple rows; revocation is one row — and it lands on live connections, not at the next reconnect.

two planes

Admin is shell, owner is device

Creating and deleting vaults, enrolling devices, exporting keys: shell-only ceremonies (centraid-gateway vault / pair / devices / key). The owner plane — your taps, grants, confirmations — rides authenticated device transport. Compromising the HTTP surface never yields landlord powers.

§03 · pairing

A ticket that burns on use.

Adding a device is a one-time ticket: minted on the desktop (Settings → Phone) or by centraid-gateway pair, rendered as a QR code, expiring in minutes. The ticket carries the gateway's dialing information, a ticket id, a one-time secret, and the vault it grants — the secret is stored only as a hash, compared in constant time, and burned on redemption.

The phone redeems it over a dedicated pairing channel on the gateway's peer-to-peer endpoint, proves the exchange, and comes out the other side enrolled: its endpoint id in the ACL, its line to the gateway permanent until you revoke it. No password, no account, nothing reusable to intercept.

Enrollment is one bit, not a role matrix — a device is full or read-only — and what it can actually see is narrowed further by each app's consent grants. That single tier plus those grants is exactly what shapes the device's offline replica, two sections down.

headless pairing · admin shell
$ centraid-gateway pair --data-dir ~/centraid-data --vault Local --ttl-minutes 15
{ "ticket": "…", "code": "…", "expiresAt": "…" } # scan or type on the phone
$ centraid-gateway devices revoke --data-dir ~/centraid-data <enrollment-id>

§04 · connectivity

Peer-to-peer QUIC, no rented middleman.

Device links run on iroh — direct, hole-punched, end-to-end encrypted QUIC between devices that authenticate by key, not by account. It replaces the expensive proprietary layer most products rent: no push relay, no sync fleet, no per-seat tunnel service.

1

One endpoint, two channels

A gateway exposes a tunnel channel — each HTTP request rides its own QUIC bi-stream, response bodies streamed so live SSE updates stay live — and a pairing channel for ticket redemption. Every connection and every stream re-checks the device allowlist, so revocation bites mid-session.

2

Transport tiers with a guardrail

Clients classify their route: local (embedded gateway, loopback), direct (an https URL + bearer — your Tailscale or LAN), iroh (dial by endpoint id; a loopback proxy makes it transparent to the rest of the app), or relay-only — the browser's lot, where a WebAssembly iroh endpoint has no UDP socket to hole-punch with, so every stream rides iroh's public relay, still key-authenticated and end-to-end encrypted. Plain http to a public hostname is refused outright — loopback and RFC1918 only.

3

Version handshake, exact-match

On connect, clients read the gateway's version and schema epoch and refuse a mismatch. Pre-release policy: exact match or walk away — compatibility windows arrive with v1, not before.

The wire protocol has one TypeScript reference implementation, ported byte-for-byte to Swift and Kotlin for the phone's native module and compiled to WebAssembly for the browser.

§05 · replica

Offline, without a second writer.

A star has one obvious weakness: cut the line to the center and the client goes dark. Centraid closes that without betraying the thesis — no second writer, no two-way merge, no CRDT. Every device keeps a consent-scoped replica: a private, offline copy of exactly the slice it's allowed to see, kept current by a one-directional change log the gateway alone writes.

The mechanism is a durable log living inside vault.db. A per-entity trigger writes a replica_change row in the same SQLite transaction as the base mutation, so the log can never disagree with the data it describes. A device never receives the whole vault: the gateway computes a server-derived shape — the app's consent grants intersected with the device's trust tier, then row- and column-minimized — and ships only that. A read-only browser session sees only its one app's shape.

1

Bootstrap, then follow

A fresh device pulls a shaped snapshot from replica/bootstrap (windowed for large libraries), then tails changes?since=<epoch:seq> — as JSON, NDJSON, or a live SSE stream — and acks the cursor it durably committed back through replica/checkpoint. Heavy columns are fetched lazily, per row, only when something opens them.

2

Writes are intents, not edits

Offline, a device doesn't mutate its copy and reconcile later — it queues an intent ({intentId, appId, action, input} with a checked payload hash) and submits it to replica/intents on reconnect. The gateway runs it once and durably; replaying the same intentId returns the same outcome instead of re-executing, so a flaky network never double-posts. The canonical order always wins — the device's optimistic view is just the log, replayed.

3

Consent still answers

An intent can come back parked — the same owner confirmation any risky command faces, now reachable from a pocket — settling later through the very log the device already follows. A read-only device gets a durable denied, server-side, not a hidden client-side guard. Revocation or downgrade lands on live connections immediately.

Compatibility here is a number, not a migration ladder: an epoch plus a build-time schema epoch are invalidation counters — bump either and clients throw the replica away and re-bootstrap clean, exactly in keeping with the pre-release no-migrations rule. This is how the star carries offline devices without ever growing a second master.

§06 · desktop shell

The everyday front door.

The Electron shell is where you live day to day: the app grid, the assistant, the automations panel with its live run stream, and Settings — vaults, phone pairing, connections, import. It embeds a gateway for the local case and dials remote ones through the same transport tiers, injecting the bearer token and vault header so iframed apps just work. Switching vaults is a client-side act: same gateway, different world.

The shell is also where parked things surface — outbox items, consent requests, confirm-gated commands — one blocking drawer instead of scattered dialogs.

§07 · web app

A client home in a browser tab.

The third home installs nothing heavier than a bookmark: the web app is a PWA that pairs with the same one-time ticket a phone uses — paste it and you're enrolled, no account, no gateway URL to hunt for — and holds the same consent-scoped replica in a service worker, so it keeps working with the tab offline.

the normal path

Relay-only iroh, in WASM

A browser owns no UDP socket to hole-punch with, so its iroh endpoint is compiled to WebAssembly and rides iroh's public relay for every stream — still key-authenticated, still end-to-end encrypted. The stable device key lives in the browser; because the transport bypasses browser HTTP entirely, the gateway stamps the trusted origin on its own sessions.

the fallback

Direct HTTP, origin-bound

Where the gateway answers over https (Tailscale, LAN), the browser can talk straight to it. The session is then an Origin-bound, HttpOnly control cookie on a dedicated origin — so logout is a server-side revoke, not just a cleared localStorage: killing the cookie kills the session on the gateway too.

§08 · mobile companion

Native where it counts, your approvals in your pocket.

The phone app (Expo, iOS and Android) embeds no gateway — but it is no longer a thin window either. Its home splits in two. Built in are three genuinely native apps — Photos, Docs and Agenda — drawn over an on-device SQLite replica, so a timeline, a folder of files, or next week's schedule all render instantly in airplane mode. Photos runs a durable background upload of the camera roll into the vault (sealed frames, an incremental hash, an Android foreground service and iOS background transfers); Agenda expands recurrence locally and fires reminders on-device, with no push broker in the loop.

Your apps — anything built on the desktop — still open in a WebView over the tunnel: the same responsive HTML, live SSE updates, search across your tiles. Building apps stays a desktop act; the phone renders, it doesn't author.

Two touches earn a mention. "Free up space" is the one and only UI that deletes phone originals, and it refuses to until the vault holds a settled copy — a custody-gated delete, never a leap of faith. And an Approvals surface carries parked commands and staged outbox items to your pocket, because consent that can only be granted at a desk is consent that gets rubber-stamped later.

Honest beta note: the native tunnel and store modules mean a dev build (bun run dev:mobile) — Expo Go won't load it, and prebuilt store binaries don't exist yet.

§09 · agent runtimes

Bring your own brain.

Chat turns, builder sessions and automation fires all run through one injected turn contract — the gateway doesn't care which agent runtime satisfies it. Two do today:

spawned

Codex

Spawned as codex app-server over JSON-RPC stdio — an on-device runner and the default brain when you haven't picked one.

in-process

Claude Code

Runs through the Claude Agent SDK inside the gateway process — the same turn contract, a different lab behind it.

There is no single "active agent" anymore — only a default and a set of routing lanes. Settings → Agents resolves each subsystem — Assistant, In-app Ask, Builder, Automations — to its own (agent, model) pair, any lane free to inherit the default. Whatever the brain, the leash is identical: agents see the vault only through vault_sql / vault_invoke / vault_content, and the model tiering is capability-based, not vendor-named — swap the runtime, keep the contract, keep every receipt.
