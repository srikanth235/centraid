## What changed

**Scope of the addition.** Today the Codex runner only talks to OpenAI's hosted models via `~/.codex/auth.json`. This adds first-class support for routing Codex through any OpenAI-compatible `/v1/chat/completions` endpoint — Ollama, vLLM, Groq, Together, LM Studio, anything that speaks the same wire format. Both chat and builder surfaces inherit it for free: the new field lives on `RunnerPrefs` and the existing `runAgentTurn` dispatcher plumbs it through. No new runner kind, no parallel tool loop — Codex remains the agent; only its model provider is swappable.

**Add `OpenAICompatProvider` + `RunnerPrefs.provider?` to `@centraid/agent-runtime/types`.** New typed shape: `{ id, name, baseUrl, wireApi?, envKey?, apiKey? }`. `claude-code` runners ignore it (the Claude Agent SDK is Anthropic-wire-format only). Type also exported from the package index so the desktop can import it for its own prefs loader.

**Materialize a scoped `CODEX_HOME` per spawn in `codex-provider-config.ts`.** `codex app-server` does not honor `-c key=value` flags (the comment at the top of `codex-app-server.ts` already documented this), and `thread/start` only accepts `model` per-call — not `model_provider` or `base_url`. So the only mechanism codex exposes for declaring a custom provider is the `config.toml` it loads from `$CODEX_HOME`. `materializeCodexHome(provider, baseDir)` writes `<baseDir>/codex-homes/<sanitized-id>/config.toml` with a `model_provider = "<id>"` line and a `[model_providers.<id>]` table declaring `name` / `base_url` / `wire_api` / `env_key`. The `apiKey` value is never written to disk — it flows separately through the spawned process's env under `provider.envKey`. Re-runs on every spawn (cheap, `fs.writeFile` is atomic from the consumer's view) so a settings change takes effect on the next turn without an explicit invalidation. `buildProviderToml` escapes TOML basic strings safely (`"`, `\`, control chars → `\uXXXX`), and provider ids that aren't bare-key-safe (regex `^[A-Za-z0-9_-]+$`) get wrapped as quoted keys. `sanitizeProviderId` produces a filesystem-safe slug for the directory name without touching the id used in the toml. Spawn env wiring lives in `codex-app-server.ts`: `buildSpawnEnv` now takes an options bag and folds `CODEX_HOME` + the API-key env var into the child env without mutating `process.env`.

**Plumb `provider` + `codexHomeBaseDir` through `runtime.ts`, `chat-adapter.ts`, `agent-session.ts`.** `AgentTurnConfig.codexHomeBaseDir?` carries the host's preferred base dir; the codex branch of `runAgentTurn` forwards `prefs.provider` + `config.codexHomeBaseDir` into the adapter. Claude branch is untouched. Both chat (`chat-adapter.ts` → `MakeChatRunnerOptions.codexHomeBaseDir?`) and builder (`builder-harness/agent-session.ts` → `CreateCentraidAgentSessionOptions.codexHomeBaseDir?`) accept the base dir from the host and pass it through. No callers needed event-shape changes — `ChatStreamEvent` is provider-agnostic.

**`probeProvider` in preflight: `GET <baseUrl>/models` with 4s abort + 401/403 handling.** New exported helper in `preflight.ts`. Uses an `AbortController` with a 4s timeout, sets `Authorization: Bearer …` only when `envKey` + `apiKey` are both set. Returns `ProviderStatus { id, baseUrl, ok, modelCount?, reason? }`. 401/403 → `"API key rejected by <id>"`; other non-2xx → `"HTTP <status>"`; abort → reason names the URL; network throw → `"failed to reach <url>: <message>"`. Counts entries in `data[]` (the OpenAI shape) when the body parses as JSON; omits `modelCount` otherwise. `runPreflight` attaches the result as `status.provider` only on the codex kind. Cache key extended with `provider.id::baseUrl::envKey` so any of those changing forces a re-probe; the API key itself isn't part of the key (rotating just the key surfaces as a 401 on the next request, which is fine).

**`ProviderStatus` on `RunnerStatus` (runtime-core).** New optional sub-status field in `packages/runtime-core/src/runtime.ts`; exported from the package index. Independent of the binary preflight — a healthy codex CLI can still fail at runtime if the endpoint is unreachable, so the settings UI surfaces both signals.

**Electron `safeStorage` for the API key (`provider-secrets.ts`) + IPC channels.** Electron `safeStorage` (macOS Keychain / DPAPI / libsecret) encrypts the plaintext into `<userData>/local-runtime/provider-key.bin`. `setProviderApiKey` / `getProviderApiKey` / `hasProviderApiKey` / `clearProviderApiKey` helpers. `ensureEncryptionAvailable()` throws when the backend is missing (Linux without `libsecret`) so the renderer's Save handler can show a clear toast instead of silently persisting plaintext. Decryption failures (e.g. OS keychain rotation) are swallowed in `getProviderApiKey` and treated as "no key" — the next request surfaces the 401 naturally rather than crashing the agent. New IPC channels: `centraid:agent:provider:setApiKey` / `:hasApiKey` / `:clearApiKey` for the safeStorage flow, plus `centraid:agent:runner:status` for a fresh re-probe of binary + endpoint. The plaintext only ever crosses the bridge in the renderer → main direction (Set). `hasApiKey` returns a boolean; `clear` and the runner-status calls don't carry secrets. Preload + `centraid-api.d.ts` updated to expose the four new methods.

**Desktop prefs loaders decrypt the key only when building `RunnerPrefs.provider.apiKey`.** `parseProviderPrefs(prefs)` in `apps/desktop/src/main/local-runtime.ts` reads `agent.runner.provider.*` from `user_prefs` and returns `Omit<OpenAICompatProvider, 'apiKey'>` — the typing enforces that the API key is never sourced from prefs. `resolveProviderPrefs(prefs)` is the async wrapper that splices in the safeStorage-decrypted key, called on every turn from both the chat-side `prefsLoader` (in `local-runtime.ts`) and the builder-side `loadRunnerPrefs` (in `ipc.ts`). New `localRuntimeCodexHomeBaseDir()` returns `<userData>/local-runtime` (sibling to `apps/` and the gateway DB). `makeChatRunner` + `createCentraidAgentSession` both receive that base dir. The `USER_PREFS_SAVE` handler also invalidates the preflight cache when any `agent.runner.*` key changes.

**Settings UI panel with presets, live endpoint status, and auto-flip to Codex.** New "Custom inference endpoint" `drawerGroup` in `apps/desktop/src/renderer/app.ts`, placed after the existing "AI providers" panel. Preset dropdown (Ollama / Groq / Together / vLLM / Custom) auto-fills id, name, baseUrl, envKey, wireApi. Text inputs for each field; the API key is a password input with status text ("A key is stored (encrypted in OS keychain)" vs "No key configured"). Live endpoint status line calls `getRunnerStatus()` on mount and after each Save/Test action; renders "Connected · N models available · `<url>`" or the failure reason. Save button writes the four `agent.runner.provider.*` keys to user_prefs (null deletes; `wireApi: 'chat'` is also elided since it's the default), sets the API key via safeStorage if the input is non-empty, and — if `agent.runner.kind` isn't already `codex` — flips it to codex in the same transaction. The toast then reads: *"Provider saved · runner switched to Codex (only OpenAI-compatible API is supported here; Claude Code is Anthropic-wire-format only)"*. Disable button clears all five prefs + the safeStorage key; it does NOT touch `agent.runner.kind` (the user's prior runner choice is preserved).

**Tests (22 new).** `codex-provider-config.test.ts` (8): toml emission shape, env_key omission for keyless providers, `wireApi=responses`, escaping of `"` / `\` / `\n`, non-bare-key quoting, on-disk materialization under `<baseDir>/codex-homes/<id>`, traversal-safety for `../escape/attempt` ids (sanitized leaf stays inside `codex-homes/`), rewrite-on-every-call (so a settings change takes effect on the next spawn). `preflight.test.ts` (14): mocked `globalThis.fetch` swaps — 200 with `data[]` returns `modelCount`, Bearer auth sent only with both envKey + apiKey, no Authorization header for keyless endpoints, URL join handles trailing slashes, 401/403 → "API key rejected by <id>", non-2xx → "HTTP <status>", non-JSON body still `ok: true`, missing `data[]` omits modelCount, network throw → "failed to reach", AbortError → reason names the URL, `runPreflight` attaches sub-status on codex with a provider configured, `runPreflight` does NOT probe on `claude-code` (the SDK ignores `provider`), and cache key includes provider id so switching id re-probes.

## Out of scope

- **Native OpenAI-compatible backend without Codex.** Would require reimplementing the tool loop, session resume, abort handling, error normalization, and the inline `centraid_sql_*` dispatch from scratch. Path A (route Codex through the endpoint) ships every OpenAI-compatible endpoint for ~a hundred lines of wiring. A native backend is only worth building if/when there's a concrete need to ship a runtime without the codex binary.
- **Mobile app.** `apps/mobile` doesn't currently consume `@centraid/agent-runtime` directly; if it gains a local-runtime path later, the same `RunnerPrefs.provider` plumbing applies without further engine work.
- **Provider-specific quirks / function-calling reliability.** The `centraid_sql_*` tools are declared as `dynamicTools` on `thread/start`; Codex translates them to OpenAI function-calling. Models that don't reliably emit tool calls (small Llamas, some Ollama defaults) will silently break the SQL pathway. This is a runtime characteristic of the chosen model, not a fix in the harness — the settings UI doesn't warn about it.
- **Migrating an existing `agent.runner.provider.apiKey` prefs entry (if any) into safeStorage.** No such entry exists in any currently-shipped build — the prefs key is new in this change.
- **Refactoring `ipc.ts` or `codex-app-server.ts` to break the 500-line cap.** Tracked via the allow-comments; both files are coherent units today and the splits should land as their own changes when the shapes settle.
- **Bringing back the legacy `codex exec` path.** `app-server` remains the only backend; this change does not affect adapter choice.
