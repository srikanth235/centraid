## What changed

**Per-app `automations.sqlite` store (runs, run_nodes, state) with lazy file creation + own migration ladder.** New `packages/runtime-core/src/automation-runs-schema.ts` defines the on-disk shape: three tables ŌĆö `runs` (run_id PK, automation_name, trigger_kind one of `scheduled|manual|replay|on_failure`, optional `parent_run_id` self-FK for sub-invocations, `input_json`, terminal `summary` + `output_json` validated against manifest `outputSchema`), `run_nodes` (CASCADE-from-runs, with `ordinal` for call order and `batch_id` shared across nodes in one Promise.all frontier), and `state` (per-(automation_name, key) KV for `ctx.state`). Indexes on `(automation_name, started_at DESC)` for the UI run list and `(name, started_at DESC)` for run-nodes tool-usage queries. Migration ladder is separate from the gateway DB ladder so the per-app file can evolve independently. `automation-runs-store.ts` wraps the file in a lazy `AutomationRunsStore` ŌĆö the DB handle isn't opened until the first method call, which means the file stays absent on disk until an automation actually fires.

The store exposes `insertRun` / `finishRun` (parent state-machine for one fire), `insertNode` (per ctx.tool/ctx.agent call with batch_id), `listRuns({name?, status?, since?, limit?})` for the UI and `ctx.runs.list` surfaces ŌĆö the `status` (`ok`/`error`) filter is applied in the SQL predicate *before* `ORDER BY ŌĆ” LIMIT`, so `{status:'ok', limit:N}` returns N successful runs even when recent failures would otherwise crowd them out of the window ŌĆö `lastRun(name, status?)` for the "since last successful run" cursor pattern, `listNodes(runId)` for the per-run timeline view, `stateGet/stateSet/stateDelete` for `ctx.state`, and `prune({count|days|errorsOnly|all})` so the manifest's `history.keep` policy can run at end-of-run with CASCADE pulling orphaned `run_nodes` along. The handle is opened with WAL + `foreign_keys=ON` for the same reasons as `data.sqlite`. (The `run_nodes.attempt` column proposed in issue #80 was dropped ŌĆö see the retry note below.)

**Manifest reshape: `trigger:{kind,expr}` canonical (v0 ŌĆö no legacy `schedule` back-compat), `outputSchema`, `onFailure`, `history.keep`.** `packages/runtime-core/src/automation-manifest.ts` grew four field surfaces. The only trigger shape is `trigger: { kind: 'cron', expr: '<cron>' }`; only `kind: 'cron'` is wired ŌĆö `webhook` / `event` will land later without a second migration. The v0 cleanup pass deleted the bare-`schedule:` fall-through (and the `AutomationManifest.schedule` mirror field): a manifest without `trigger` is now a hard parse error. All bundled templates and the mirror-table writer (`AutomationStore.upsert` ŌåÆ `manifest.trigger.expr`) were updated to match. Existing consumers read `row.cronExpr` from the storage row, not the manifest, so the change is contained.

`outputSchema` declares the shape of the handler's optional `return { summary, output }`. The validator accepts the JSON-Schema subset that issue #80 calls out (object root, scalar/object/array property types, `required`). `validateOutputAgainstSchema(schema, output)` returns `null` on pass or a human-readable error string on fail; the runtime feeds that into `runs.error` + `runs.ok=0` when the shape doesn't match. Full JSON-Schema support stays out of scope.

`onFailure` names a sibling automation to dispatch on handler failure. The validator enforces `isValidAutomationName` shape; cross-reference validation (the named automation actually exists) happens at sync-time in `sync-automations.ts` rather than at manifest-parse time, since the parser sees one manifest at a time.

`history.keep` is one of `{count: N}` | `{days: N}` | `"all"` | `"errors"`, defaulting to `{count: 100}` when omitted. The retention shape is parsed here so the runtime can hand it straight to `AutomationRunsStore.prune` at end-of-run; the prune predicate itself was already wired in the previous commit.

**Instrument `runAutomationHandler` to commit `runs` + `run_nodes` rows around handler execution (incl. batch_id).** The runner takes a required `runsStore: AutomationRunsStore` plus `triggerKind`, `input`, `parentRunId`, `outputSchema`, and `history`. On run start it writes the `runs` row with `trigger_kind` defaulting to `'scheduled'`. Every `ctx.tool` batch from the worker mints one batch_id when N > 1 (solo calls get `batchId: null` so the timeline UI can tell them apart) and one `ordinal` per call; the dispatcher result lands as a `run_nodes` row. Every `ctx.agent` call is its own ordinal as a separate `kind: 'agent'` row. At end-of-run the `runs` row is updated with `ok`, `error`, `summary`, `output_json`, and retention runs against the manifest's `history.keep`. Args + output JSON above 64 KB are replaced with a `{_truncated: true, bytes, head}` envelope so a tool that returns 50 KB of PRs serialized per fire doesn't balloon the file. The v0 cleanup also dropped the bare-string-as-summary back-compat in `extractReturnEnvelope` ŌĆö handlers now must return `{ summary, output }` (or undefined) for the summary field to populate.

**`ctx.state.{get,set}`, `ctx.runs.{last,list}`, `ctx.invoke(name, {input})` against the per-app file.** Worker side: each method flushes any pending tool batch first (these are different turn shapes so they don't ride along), then round-trips through new `state`/`runs`/`invoke` workerŌåÆparent message types. Parent side: `state.get` returns the JSON-parsed value (falling back to the raw string if it isn't valid JSON), `state.set` accepts any JSON-serializable value (undefined ŌåÆ null), `state.delete` is a single-row drop. `runs.last({name?, status?})` and `runs.list({name?, status?, since?, limit?})` query the per-app store; the in-progress self-run is filtered out so the handler doesn't see its own incomplete row. `ctx.invoke` routes through a new host-supplied `invokeDispatcher` callback the handler-runner takes as a constructor parameter ŌĆö runtime-core can't load and execute a sibling automation by name, but it can hand the child run's parent_run_id back via the callback so the host's recursive `runAutomationHandler` call links the audit chain.

**`ctx.tool` failure handling ŌĆö no runtime retry knob (community-critique resolution).** Issue #80 proposed `opts.retry: { max, backoff }` + `opts.onError` as a third argument to `ctx.tool`. A community critique pointed out the flaw: a flat `{ max, backoff }` knob set at the call site is taxonomy-blind ŌĆö it retries a 404 as eagerly as a 429, and the audit rows just record the same guess being wrong N times. The deeper issue is that centraid's tools are arbitrary MCP servers; MCP hands back `isError: true` + an opaque string, so there's no error taxonomy to attach per-class policy to. And issue #80's own design test ŌĆö *"does this solve a problem a JS-handler can't solve itself?"* ŌĆö kills the knob: the handler is plain JavaScript, retry-with-classification is `try/catch` + `if (err.message.includes(...))` + a sleep, and only the handler knows the tool's error semantics. So `opts.retry` / `opts.onError` were dropped before landing: a failed `ctx.tool` rejects the Promise, the handler owns retry. The vestigial `run_nodes.attempt` column went with it; handler-driven retries surface as distinct ordinals (they *are* distinct calls). The system prompt teaches the `try/catch` + classified-retry pattern with a worked `withRetry` helper instead.

Outsized growth of `automation-handler-runner.ts` is held in check by a new `automation-handler-audit.ts` (truncation, retention, envelope extraction, node-row writers) and `automation-handler-ctx.ts` (the `ctx.state` / `ctx.runs` / `ctx.invoke` message handlers + tool-batch dispatcher). The runner itself just owns Worker lifecycle, the timeout/abort plumbing, and the message router.

**`onFailure` dispatch (incl. timeout/crash) with depth-3 recursion cap.** Both hosts now own the failure cascade. `packages/agent-runtime/src/run-automation-local.ts` (local OS-scheduler path) and `packages/openclaw-plugin/src/lib/automations-provider.ts` (remote openclaw cron path) each opt-in to the new ctx surface by constructing an `AutomationRunsStore` keyed off the app's data dir, wiring an `invokeDispatcher` that recursively re-enters the host (intra-app only, per issue #80 ┬¦ Out), and inspecting the handler outcome at end-of-fire. When `outcome.ok === false` and `manifest.onFailure` names a sibling automation, the host fires it with `triggerKind: 'on_failure'`, `parentRunId` linked to the failed run, and an input payload carrying the failed runId, automation name, error message, and node summary (ordinal/kind/name/ok/error). A `failureDepth` counter threaded through the recursive call refuses to dispatch beyond depth 3, logging an `aborted at depth N (cap=3)` warning instead ŌĆö that's the issue's "Open question ŌĆö onFailure recursion" resolution. Timeout-triggered failures and worker crashes go through the same path because `runAutomationHandler` always returns an outcome with `ok=false`/`error` set, regardless of the failure mode.

**`outputSchema` validation of handler return; failures land in `runs.error`, `runs.ok=0`.** Wired in the previous commit's `runAutomationHandler` change. After the handler returns successfully, the runner extracts the `{summary, output}` envelope and validates `output` against `manifest.outputSchema` via `validateOutputAgainstSchema`. A validation failure flips the outcome to `ok=false` with a `outputSchema validation failed: <reason>` error message; the audit row sees `ok=0` and the error. Five new agent-runtime tests cover the local path end-to-end including this rejection.

**Retention pruning at end-of-run per `history.keep`.** Also threaded in the previous commit. The runner reads `manifest.history.keep` and calls `AutomationRunsStore.prune(name, { count | days | errorsOnly | all })` after `finishRun`. CASCADE on the `run_nodes` foreign key drops the orphaned node rows so the file stays bounded. Default `{count: 100}` applies when the manifest doesn't declare a policy.

**Desktop UI: per-automation run list + per-run node timeline.** Two new IPC channels (`AUTOMATIONS_LIST_RUNS`, `AUTOMATIONS_LIST_RUN_NODES`) opened in `apps/desktop/src/main/ipc.ts` read directly from the per-app `automations.sqlite` and return [] when the file doesn't exist yet (no automation has fired). Preload surfaces them as `window.CentraidApi.listAutomationRuns({appId, name, limit})` and `listAutomationRunNodes({appId, runId})`. The Standing Orders card grew a "Runs" toggle in its foot; clicking it lazy-loads the last 25 runs inline below the card with timestamp + ok/Ō£Ś icon + triggerKind + duration + summary, newest first. Each row is a button that expands to show the node timeline ŌĆö ordinal (`#0`, `#1┬Ę2` for retries), kind (TOOL / AGENT), name, batch_id, duration, plus `<details>` blocks for args/output JSON. Styles live next to the existing `.cd-app-order-*` block in `styles.css`. Error-state rows wear a red border + danger background; node rows that hit the truncation envelope still render the head + size ŌĆö the user just doesn't get the full payload back.

**System-prompt `### Run audit & state` block under `### Automations`.** Appended to `packages/builder-harness/src/system-prompt.ts`. Teaches the builder agent: (1) `ctx.tool`/`ctx.agent` are auto-instrumented ŌĆö no manual logging; (2) `ctx.state.get/set` for cursors/ETags/dedup hashes; (3) `ctx.runs.last({status: 'ok'})` for the since-last-successful-run pattern; (4) `ctx.runs.list({since})` for aggregating + catch-up patterns; (5) there is no retry knob ŌĆö a failed `ctx.tool` rejects the Promise and the handler does classified retry with `try/catch` (a worked `withRetry` helper is shown); (6) `ctx.invoke(name, {input})` for sub-invocations; (7) `return { summary, output }` is validated against `manifest.outputSchema`. Calls out the four pattern names ŌĆö stateless poll, incremental, aggregating, catch-up ŌĆö and explicitly directs the builder away from inventing `runs` / `watermark` tables in `data.sqlite`. The manifest example in the same prompt now shows the new `trigger`/`outputSchema`/`onFailure`/`history.keep` fields.

**Boundary test: `centraid_sql_*` agent tools cannot reach `automations.sqlite`.** Five new tests in `automation-runs-boundary.test.ts` assert that `describeOp({dataFile: data.sqlite})` does not list `runs`/`run_nodes`/`state` even when `automations.sqlite` has been populated; `readOp` errors on SELECTs against those tables (they don't exist in `data.sqlite`); and a side-channel probe via `SELECT name FROM sqlite_master` confirms the audit tables aren't visible. Together these establish that `centraid_sql_*` agent tools ŌĆö which take `dataFile` and never see the `automations.sqlite` path ŌĆö cannot read or write the audit/state surface.

**Unit tests across runtime-core / agent-runtime; typecheck + lint green.** Counts after the follow-up + PR-review fixes: runtime-core 288, agent-runtime 96, openclaw-plugin 21, builder-harness 4, chat-harness 4 ŌĆö 413 total. Typecheck 16/16. `bun run check` (oxfmt --check + oxlint) clean.

**Pinned-data replay ŌĆö `runs.pinned` + `triggerKind: 'replay'` serve recorded `run_nodes` for builder iteration.** Re-running an automation while iterating on its handler hits live MCP tools (GitHub, Linear, ŌĆ”) on every fire ŌĆö slow and non-deterministic. Migration `1 ŌåÆ 2` adds `runs.pinned` (a run flagged as a replay fixture); `AutomationRunsStore` grows `setPinned` / `pinnedRun`, and all three `prune` predicates gain `AND pinned = 0` so a pinned fixture is never dropped by retention. `runAutomationLocal` takes a `replayFromRunId` option: when set it forces `triggerKind: 'replay'`, skips the mock-LLM server + CLI spawn entirely, and routes `ctx.tool` / `ctx.agent` / `ctx.invoke` through `buildReplayDispatchers` (new `run-automation-replay.ts`), which serves the pinned run's recorded `run_nodes` matched by call **name** (required) and then preferring an exact serialized-args match ŌĆö so a repeated call/invoke of the same target with a different payload replays its own recorded node, and a name miss fails loudly rather than serving an unrelated node's output. `runAutomationLocal` closes the audit store it owns in a `finally` (a caller-injected store stays the caller's to close), so a long-lived host doesn't leak a SQLite handle per fire. The live `ctx.tool` / `ctx.agent` dispatch + mock-server lifecycle moved to `run-automation-live-dispatch.ts` so `run-automation-local.ts` stays under the 500-line cap and the replay vs. live branch is legible.

**Cross-app `ctx.invoke` ŌĆö `ctx.invoke('appId/name')` resolves any registered app.** `ctx.invoke` now accepts an `appId/automation` target on **both** runtime paths. Local: `runAutomationLocal` parses the prefix and resolves a non-local `appId` through a host-supplied `resolveApp` callback (the desktop wires it to the local apps dir). Openclaw remote: `runOpenclawFire` resolves it through the provider's existing `resolveAppDir` hook ŌĆö the same `runtime.registry` lookup the desktop and mobile clients use to list apps ŌĆö so cross-app invoke needs no new gateway surface. On both paths the cross-app child runs against the *target* app's own `automations.sqlite` and is **not** linked via `parent_run_id` (the self-FK can't cross SQLite files); intra-app children keep the link. The openclaw per-fire logic (`runOpenclawFire`, the three dispatchers, the config ref) moved to a new `openclaw-fire.ts` so `automations-provider.ts` stays under the 500-line cap.

**`ctx.invoke` recorded as a `kind: 'invoke'` audit node; `ctx.input` exposed to handlers.** `handleInvokeMessage` now records every `ctx.invoke` as a `run_nodes` row of new kind `'invoke'`, carrying the invoked target name and (migration `1 ŌåÆ 2`'s new `run_nodes.child_run_id`) the spawned child run id ŌĆö the DAG view nests the child from this link, and `AutomationRunsStore.listChildRuns` surfaces the same linkage by `parent_run_id`. `AutomationInvokeDispatcher` returns `{ output, childRunId }` so the node can be linked. Separately, the payload passed to `ctx.invoke(name, { input })` (and the `onFailure` failure summary) is now delivered to the handler as `ctx.input` ŌĆö the worker threads `workerData.input` onto the `ctx` object ŌĆö so an invoked automation can actually read what it was called with.

**Available-tools grounding block ŌĆö host tools captured with exact JSON schemas via the mock-LLM server.** Every coding agent ships its LLM, on the very first request, the complete callable tool set ŌĆö builtins *and* MCP tools ŌĆö each with a full JSON input schema; that is the agentŌåömodel contract. So `enumerateHostTools` (agent-runtime, `host-tools.ts`) points the CLI at the same per-run mock-LLM server the automation runtime uses (`startMockLlmServer`, new `onRequest` hook), stages an immediate end-turn, spawns the runtime via `defaultSpawnCli`, and snapshots the `tools` array off that first request ŌĆö aborting the CLI the moment it lands, so zero model tokens are spent. This *unblocks codex builtins*: the earlier `mcpServerStatus/list` route saw only MCP tools (codex has no builtin-enumeration API), whereas the request payload carries everything. codex's OpenAI Responses `tools` (`{type:'function', name, parameters}` / `{type:'custom', name}` / `{type:'web_search'}`) and claude's Anthropic Messages `tools` (`{name, description, input_schema}`, MCP tools as `mcp__server__tool`) both normalize to a flat `HostTool[]` (`{name, source, server?, description?, inputSchema?}`). This is deliberately *the same path* the automation runtime drives, so the enumerated surface is exactly what a deployed handler's `ctx.tool` can reach ŌĆö not a second-hand registry that might over- or under-promise. `buildToolsGroundingBlock` renders an `### Available host tools` section listing every tool with a `_(native)_` / `_(mcp)_` tag, its description, and its exact JSON args schema, plus the `ctx.tool` / `requires.tools` / `requires.mcps` rules; `agent-session.ts` splices it in at session start. Best-effort ŌĆö a missing/old CLI or a probe timeout yields no block (the degrade-silently pattern of the live-schema block).

**codex 0.128 / claude 2.1 CLI compatibility ŌĆö Responses API + corrected exec flags.** Driving the current CLIs surfaced three stale assumptions in the runtime: codex 0.128 rejects `wire_api = "chat"` at config load (Responses API only), `codex exec` no longer accepts `--ask-for-approval` or `--allowed-tools`, and `claude --print --output-format=stream-json` now requires `--verbose`. The mock-LLM server gained a third wire protocol ŌĆö `writeOpenAiResponses` (`mock-llm-writers.ts`) emits the Responses SSE event stream (`response.created` ŌåÆ `response.output_item.{added,done}` ŌåÆ `response.completed`), with tool calls as `function_call` output items; `extractToolResults` learned to read `function_call_output` items from the Responses request `input` array. `codex-provider-config.ts` now defaults `wire_api` to `responses`; `run-automation-cli-spawn.ts` drops the invalid codex flags (using `--dangerously-bypass-approvals-and-sandbox` + `--skip-git-repo-check` ŌĆö the manifest tool allowlist is enforced by the mock staging only permitted calls, so `codex exec` needs no allowlist flag) and adds `--verbose` to the claude invocation; `SpawnCliInput` grew an optional `binPath`. Verified end-to-end: a staged-tool-call round-trip drove codex 0.128.0 through execute ŌåÆ `function_call_output` ŌåÆ ack.

**OpenClaw host-tool enumeration via `openclaw agent --local` + mock-LLM server.** The mock-capture method generalizes to a third runtime. OpenClaw runs its deployed automation agent in-process (`runEmbeddedAgent`), so there is no spawned CLI on the *deploy* path ŌĆö but enumeration never uses the deploy path: it spawns a throwaway single-turn probe, exactly as codex/claude do. OpenClaw's probe entry point is `openclaw agent --local` (embedded agent, no running gateway), and its mock base URL is wired through a *config file* rather than an env var: a generated `openclaw.json` declares a `centraid-mock` model provider (`models.providers.<id>.baseUrl` ŌåÆ the mock, `api: anthropic-messages`), and `OPENCLAW_CONFIG_PATH` / `OPENCLAW_STATE_DIR` / `OPENCLAW_HOME` point OpenClaw at an isolated probe home ŌĆö the user's real `~/.openclaw` is never read or written. New `openclaw-tool-probe.ts` owns the config builder (`buildOpenClawProbeConfig`), the home materializer (`materializeOpenClawProbeHome`), and the spawn (`spawnOpenClawProbe`). `host-tools.ts` `enumerateHostTools` now takes a `ToolHostKind` (`RunnerKind | 'openclaw'`) ŌĆö the `openclaw` branch drives `spawnOpenClawProbe`, and because the provider speaks `anthropic-messages` the captured `tools` array normalizes through the existing `normalizeClaudeTools`. The builder's `RunnerKind` stays `codex | claude-code` and `LocalRunnerKind` stays narrow ŌĆö the automation runner never spawns OpenClaw; the probe is enumeration-only. Verified end-to-end: `enumerateHostTools('openclaw')` returned all 28 OpenClaw tools with full JSON input schemas (`exec`, `web_search`, `cron`, `sessions_spawn`, ŌĆ”).

**Desktop per-run DAG view ŌĆö batched lanes + nested child-run timelines + pin/replay controls.** The flat node list in the Standing Orders runs panel became a DAG: `renderNodeTimeline` groups nodes sharing a `batchId` (a `Promise.all` frontier) into a parallel lane shown as a labelled `parallel ├ŚN` cluster, and `kind: 'invoke'` nodes get an expand toggle that lazy-loads the child run's own timeline (recursively, depth-capped) via the existing `listAutomationRunNodes` IPC. Cross-app invoke nodes render a note instead of a dead toggle (the child lives in another file). Each run row grew a Pin / Unpin button (new `AUTOMATIONS_PIN_RUN` IPC ŌåÆ `setPinned`); when any run is pinned, the panel shows a "Replay pinned run" bar that calls `runAutomationNow` with `replay: true`. Styles live next to the existing `.cd-app-run-*` block.

**Chat store consolidation ŌĆö ChatStore + ChatHistoryStore merged into one central SQLite store.** Chat state lived in two places: a per-app `_chat/` folder (`ChatStore` ŌĆö `index.json` window metadata + adapter-resume ids + a `w<id>.jsonl` transcript) and the central gateway SQLite (`ChatHistoryStore` ŌĆö `chat_sessions` / `chat_messages`). But a chat can reference more than one app, so it was never genuinely app-scoped ŌĆö the per-app folder was the wrong home, and the `_chat/*.jsonl` transcript was a second, lossy reconstruction of what `chat_messages` already holds. `ChatStore` and the `_chat/` folder are deleted; all chat state lives in the gateway SQLite, and a chat session IS the chat window (the session id is the window id). `gateway-db.ts` reshapes the baseline schema: `chat_sessions` drops `NOT NULL app_id` for a nullable `origin_app_id` (the app the chat was opened from; `NULL` = started from the centraid shell), and gains a sticky `mode`, the runner-resume handle (`adapter_kind` + `adapter_session_id`, formerly `ChatStore`'s `index.json`), and `turn_count`; `chat_messages` gains a nullable per-message `app_id` (which app a tool call in that message touched ŌĆö the grain a future cross-app token-cost rollup attributes against). `ChatHistoryStore` absorbs `ChatStore`'s surface ŌĆö `noteTurn` (bump `turn_count` + persist the resume handle), `getSessionMeta`, `mode` + `origin` on `createSession`. Surface A (`/centraid/<id>/_chat`) is now POST-only: the `windows` / `history` / `delete` routes are gone (Surface B already lists / loads / deletes), and the route reads sticky mode + resume handles off the session row, threads the previous handle to the runner via `ChatRunInput.prevAdapter*`, and records completion via `noteTurn`. The runner-owned session file (the OpenClaw pi `SessionManager` resume file ŌĆö codex / claude resume by id alone) moved out of the app folder to a central `chatRunnerSessionDir` scratch path. `chat-adapter.ts` drops its `ChatStore` lookup; `chat-harness` loses the now-dead `listChatWindows` / `fetchChatHistory` / `clearChatWindow` window helpers.

**Run audit relocated into the central gateway DB so cross-app `ctx.invoke` links one DAG.** The automation run audit (`runs`, `run_nodes`, `state`) and `ctx.state` lived in a per-app SQLite file (`<appDir>/automations.sqlite`). That broke the cross-app `ctx.invoke` run DAG: a child run of `ctx.invoke('appId/name')` landed in the *target* app's file, and the `parent_run_id` self-FK can't cross SQLite files, so the run tree was unjoinable. All three tables now live in the central gateway DB (`centraid-gateway.sqlite`) as new `gateway-db.ts` `MIGRATIONS[2]` ŌĆö `automation_runs`, `automation_run_nodes`, `automation_state` ŌĆö taking the migration count from 2 to 3. `automation_runs.parent_run_id` carries `ON DELETE SET NULL` so deleting one app's runs can't FK-fail against another app's cross-app child; `automation_run_nodes` cascades off `automation_runs`. `automation_runs` and `automation_state` carry a nullable `origin_app_id` (forward-compat for app-less automations, not built in v0 ŌĆö today it is always set). `automation-runs-schema.ts` is now pure types (the `AUTOMATIONS_DB_FILE` / `AUTOMATIONS_MIGRATIONS` / `migrate` / `openAutomationsDb` / `automationsDbPath` surface is deleted); the DDL lives in `gateway-db.ts`. `AutomationRunsStore` is reworked onto the shared `DatabaseProvider`: `new AutomationRunsStore(provider, originAppId)`, name-scoped reads/writes carry an `origin_app_id IS ?` predicate, `forApp(otherId)` returns a sibling store sharing the connection (used by cross-app invoke so the child run is recorded under the target app), `deleteAppData()` drops one app's run audit on deregister, and `close()` is a no-op (the gateway connection is host-owned, shared with `UserStore` / `ChatHistoryStore` / `AutomationStore`). The prepared-statement block + raw-row mappers moved to `automation-runs-store-sql.ts` to keep the store under the 500-line cap. Both runtime paths ŌĆö agent-runtime's `run-automation-local.ts` and the openclaw plugin's `openclaw-fire.ts` ŌĆö now always pass `parentRunId` for a cross-app invoke (the DAG fix) and use `runsStore.forApp(targetAppId)` instead of constructing a second per-file store; `runAutomationLocal` takes a `gatewayDb?: DatabaseProvider`, threaded through the recursive invoke + onFailure calls and through the desktop IPC, the openclaw provider, and `centraid-cli.ts` (which resolves `CENTRAID_GATEWAY_DB ?? <appDir>/centraid-gateway.sqlite`). The OS scheduler bakes `CENTRAID_GATEWAY_DB` into the launchd plist / systemd unit / Task Scheduler artifact (new `OsSchedulerJobSpec.env` + `OsSchedulerHostOptions.gatewayDbPath`) so a desktop-scheduled fire writes its audit to the gateway DB the desktop UI reads. Separately, the `automations` mirror's origin-app column was renamed `app_id` ŌåÆ `origin_app_id` and `AutomationRow.appId` ŌåÆ `originAppId` consistently through every consumer. The app-delete cascade in `runtime.ts` now also drops the app's run audit + `ctx.state` from the gateway DB (best-effort, alongside the existing mirror cleanup).
