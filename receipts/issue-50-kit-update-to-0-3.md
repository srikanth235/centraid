## What changed

**Runtime files re-stamped to kit-version 0.3.** `.governance/run.sh`, `.governance/lib.sh`, and the kit-owned workflow `.github/workflows/governance.yml` carry no behavior changes between 0.2 and 0.3 beyond the marker bump. The version pin moves forward but the discovery loop, lib helpers, and workflow body are byte-equivalent. The marker rewrite is still load-bearing: `governance kit update` reads the per-file `kit-version=` token as the source of truth for drift detection (`install.yaml.kit_version` is a cache that mirrors it), so leaving the markers at 0.2 would have made the next run think we hadn't actually updated.

**`actions/checkout` pinned to SHA in governance.yml.** Upstream 0.3 swapped `actions/checkout@v5` for `actions/checkout@93cb6efe18208431cddfb8368fd83d5badbf9bfd  # v5.0.1`. SHA-pinning third-party Actions is a supply-chain posture: tag references can be re-pointed after the fact, SHAs cannot. The companion `# v5.0.1` annotation keeps the file readable for humans without weakening the pin.

**`setup-clone.sh` renamed to `enable-governance.sh`.** Same script, just renamed throughout (file path, internal echo labels). Done as a `git mv` so blame/history follows the file. No content change beyond the user-visible echo strings.

**Hook dispatchers regenerated by hooks.sh.** All five hooks under `.githooks/` (`pre-commit`, `commit-msg`, `prepare-commit-msg`, `post-commit`, `pre-push`) re-emitted via `generate_hooks_for_strategy githooks 0.3`. The dispatcher bodies are framework-agnostic and scan installed directives at runtime, so the regenerate is mostly a marker bump — except for `pre-commit`, which needs the lint-guard preservation step below.

**Lint-guard block from #19 preserved in pre-commit.** The 0.2 `pre-commit` carried a marked sidecar block that calls `scripts/lint-staged.sh` before the governance dispatch. The block itself anticipates kit-update regeneration ("If a future `governance kit update` regenerates this hook and drops this block, re-apply it from scripts/lint-staged.sh") — I captured the block before regenerating, then re-injected it at the same anchor (right after the `GOVERNANCE_DIR=…` line). Net diff on `pre-commit` is the marker bump + the regenerated body; the lint guard reads identically.

**install.yaml field renames applied.** `version: "3"` was already correct, but two fields had pre-rename names: `agents_md_directive` (→ `agents_md_snippet`) and `setup_clone_script` (→ `enable_governance_script`). The 0.3 install schema treats these as the canonical names, and `kit update` reads `enable_governance_script` to find the Path-A enable script. Without the rename, the next `kit update` run would see the destination field as absent.

**AGENTS.md link updated for the rename.** Line 36 pointed at the old `scripts/setup-clone.sh`; rewrote both the visible text and the href to `scripts/enable-governance.sh`. Caught by the `no-broken-internal-doc-links` directive on the first smoke-test pass.

**Smoke test re-run after rename fix.** Initial `bash .governance/run.sh` post-sync flagged the AGENTS.md link as broken (rename ripple). After fixing the link the second smoke run shows 12 directives pass and only the pre-existing `commit-message-format` violation on HEAD remains — that one is commit `49e7929` (101-char subject from #49 merge), not introduced here.

**Core pack re-pinned to upstream SHA 18098b6.** Second commit on this issue (after the kit-runtime sync landed at `1c8a531`). The lockfile now records `governance-kit/core` at version `0.3`, ref `gh:Duaility/governance-kit/packs/core@main`, sha `18098b662377093182186f67a1ab0c6a6ab610d7`, `min_governance_kit: 0.3`. Re-pin done via `packverb lock-add` which upserts by id — the prior `0.2 / 1d3e8d18` entry is replaced atomically, no manual lockfile edits.

**All 13 installed directive folders overwritten with 0.3 content.** Re-fetched the pack from cache, then replaced each directive's `check.sh`, `constitution.md`, `directive.yaml`, and `hooks/` contents in place. Biggest diff was `required-docs` (now supports a per-sub-check waiver via `<!-- governance: allow-required-docs <sub-check> <reason> -->` in CONSTITUTION.md — backward-compatible enhancement, no new mandatory docs). Other directives gained minor refinements to error messages and edge-case handling. Post-update smoke test: `bash .governance/run.sh` → all 13 pass.

**evals/ and install-assets/ excluded from the target-repo install.** Upstream 0.3 ships per-directive `evals/test.sh` and `install-assets/` subdirectories. The kit's own `install_directive_folder` / `copy_tree_without_evals` helpers in `assets/packs/lib/install.sh` explicitly skip these — evals are written for the governance-kit repo's own CI (their `ROOT="$(cd ... ../../../../.. && pwd)"` resolves to the kit repo, not this one) and install-assets are consumed once at install-time and copied elsewhere, not tracked under `.governance/packs/`. Matched the 0.2 install policy.

**workflows-hardened directive skipped intentionally.** Upstream 0.3 added a new `workflows-hardened` directive and promoted it into the `minimal` preset. Per the `pack update` contract, the verb does not auto-install new directives — that's an explicit `directive add` (or `pack add` against a preset). User opted to keep the current 13-directive surface; can be installed later with `governance directive add workflows-hardened`.
