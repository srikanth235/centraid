## What changed

**Scope of the partial implementation.** Cron was wired across five packages and the desktop/mobile apps, but no path actually ran a scheduled job. runtime-core declared a `Scheduler` interface, a reconciler that walked each app's `crons/` directory, per-cron bearer tokens persisted in `_registry.json`, a `cronStatus` mirror on each `RegistryEntry`, a `CronModule` shape, the `'cron'` `handlerKind`, and the routes `/centraid/<id>/_crons`, `/centraid/<id>/_crons/<cron>/run`, `/centraid/<id>/_ingest/<cron>`. openclaw-plugin shipped `OpenClawScheduler` (CLI by default; opt-in path A via `ctx.getCron()`), forwarded `gateway_start` to bootstrap and `cron_changed` to a status mirror. desktop's local-runtime defaulted to `NullScheduler`, which logged a one-shot warning and silently dropped every registered job. builder-harness scaffolded a `crons/` directory and the system prompt taught the agent to author cron modules. The OpenClaw CLI path was the only one that could ever fire ‚Äî and it depends on a remote OpenClaw gateway that the desktop doesn't run. Rather than carry this dark surface, strip it all out.

**Delete runtime-core cron modules (`scheduler.ts`, `cron-sync.ts`, `null-scheduler.ts`, `payload.ts`).** The `Scheduler` interface + `CronJobDefinition` / `CronJobSnapshot` / `CronChangedEvent` types lived in `scheduler.ts`. `cron-sync.ts` housed the `CronSync` reconciler that walked `crons/` per app and minted webhook tokens. `null-scheduler.ts` was the local-mode default that discarded jobs. `payload.ts` parsed the OpenClaw cron-webhook payload for the ingest handler. All four files are deleted outright; no replacement.

**Strip cron fields, methods, and route kinds from `runtime.ts`, `registry.ts`, `router.ts`, `route-handlers.ts`, `types.ts`, `handler-runner.ts`, `worker/runner.ts`, `log-store.ts`, `security.ts`, `app-paths.ts`, `change-bus.ts`, `index.ts`.** `RuntimeOptions` no longer takes `scheduler` or `gatewayBaseUrl`. `Runtime.setScheduler`, `Runtime.setGatewayBaseUrl`, `Runtime.onCronChanged`, and the `cronSync` field are removed; `bootstrap()` only loads the registry and recovers torn `current.json` files. The router drops the `app-crons-list`, `app-cron-runnow`, and `app-ingest` `Route` kinds and the `/_crons` and `/_ingest` URL branches; `Runtime.handle` no longer has the matching `case` arms. `RouteContext` loses its `cronSync` field and `handleAppIngest` is deleted. `RegistryEntry` loses `cronTokens` and `cronStatus`; `Registry` loses `mintCronToken`, `cronToken`, `forgetCron`, `setCronStatus`, the `unknown_app` error code, and the now-unused `crypto` import. `CronStatus`, `CronModule`, `CronHandler`, `CronHandlerArgs` go from `types.ts`. `handlerKind` and the log-store `source` union narrow from `'query' | 'action' | 'cron'` to `'query' | 'action'` (handler-runner, worker/runner, log-store). `RESERVED_DIRS` in `security.ts` drops `'crons'`. Doc-comment cron references in `app-paths.ts` and `change-bus.ts` are tidied. `index.ts` drops the `NullScheduler`, `Scheduler`, `CronJobDefinition`, `CronJobSnapshot`, `CronChangedEvent`, `CronHandler`, `CronHandlerArgs`, `CronStatus`, `CronModule` exports.

**Drop the loopback ingest gate from `http-server.ts` + `http-utils.ts`.** `startRuntimeHttpServer` previously carved out `POST /_ingest/<cron>` from the bearer check (it was gated separately by per-cron tokens + loopback). With ingest gone, every request now requires the server-level bearer. The `isIngestRequest` helper inside `http-server.ts` is deleted, as are `isLoopback` and `getHeader` from `http-utils.ts` ‚Äî none of them had other callers. The token-field doc on `RuntimeHttpServerOptions` is updated to drop the "non-loopback-ingest" wording.

**Delete `openclaw-plugin/src/lib/openclaw-cron.ts`; rewrite `src/index.ts` to drop the `Scheduler` wiring and the `cron_changed` event handler.** `OpenClawScheduler` (and the `CronService` / `CronCreateInput` shape it depended on) is gone. `src/index.ts` constructs `Runtime` without a `scheduler` or `gatewayBaseUrl`, runs `runtime.bootstrap()` from `gateway_start` directly (no `ctx.getCron()` upgrade path), and no longer listens for `cron_changed`. The re-export of `OpenClawScheduler` and the `CronHandler` / `CronHandlerArgs` type re-exports are dropped from the public surface.

**Update openclaw-plugin manifests (`package.json`, `openclaw.plugin.json`) + README to drop the cron surface.** `package.json#description` and `openclaw.plugin.json#description` lose the "cron-fed ingest" tail. `openclaw.plugin.json#configSchema` removes the `gatewayBaseUrl` property entirely (it only existed to point webhooks at the gateway). README sections strip: the `/_crons` and `/_ingest` rows from the URL surface table; the cron handler example block; the "App folder layout" `crons/<name>.js` line; the `CronHandler` / `CronHandlerArgs` rows from the handler-type table; the `gatewayBaseUrl` config row; the "Cron registration" section in full; the per-cron bearer + ingest loopback bullets from the security model; and the entire "Open items" appendix that called out the `PluginHookGatewayCronService` shape mismatch and the webhook payload parsing heuristics.

**Remove `setLocalScheduler` / `NullScheduler` plumbing from desktop's `local-runtime.ts`.** The exported `setLocalScheduler(s: Scheduler)` setter and its `scheduler` module-level slot are gone, along with the `NullScheduler` import and the fallback that constructed one when no scheduler was set. `Runtime` is now constructed with just `{ appsDir, userStore, chatHistoryStore, logger }` ‚Äî no scheduler, no `gatewayBaseUrl`. The post-`listen` `runtime.setGatewayBaseUrl(server.url)` call is removed (the URL was only needed to construct cron webhook targets).

**Drop `crons`/`cronStatus` from desktop renderer types and builder backend-dirs set; drop `'cron'` from the log-source union.** `apps/desktop/src/renderer/centraid-api.d.ts` narrows `CentraidLogEntry.source` from `'query' | 'action' | 'cron'` to `'query' | 'action'` (both the module-level and the global-declaration occurrences). `apps/desktop/src/renderer/builder.ts` drops `'crons'` from the `BACKEND_DIRS` set used to bucket project sub-folders and updates the Logs panel subtitle from "query, action, and cron handlers" to "query and action handlers".

**Drop `crons`/`cronStatus` from mobile's `AppRegistryRow` consumers.** `apps/mobile/src/lib/gateway.ts` removes the two fields from the `AppRegistryRow` interface and tweaks the surrounding comment so it no longer mentions cron state. `apps/mobile/src/screens/AppDetail.tsx` drops the cron fields from the fabricated `AppRegistryRow` passed to `resolveAppMeta`.

**Strip cron handling from builder-harness: `scaffold.ts` no longer creates `crons/`, `clone.ts` and `hasAnyBuiltJs` no longer walk it, `system-prompt.ts` drops the cron section + ingest rules, `gateway-client.ts` + `types.ts` drop cron fields.** `scaffoldProject` no longer `mkdir`s `crons/`; the README template emitted for new projects loses the `crons/<name>.js` row. Both `scaffold.ts#hasAnyBuiltJs` and `clone.ts#hasAnyBuiltJs` walk only `queries` and `actions` now. `system-prompt.ts` loses the cron line in the folder layout, the entire `crons/<name>.js` example block, the "Cron handler design rules" section, and the "Cron webhook ingest is loopback-only‚Ä¶" bullet in the security model. `gateway-client.ts#AppRegistryRow` and `types.ts#ProjectInfo` drop their cron fields. The `built` doc-comment narrows accordingly. The package README's `scaffoldProject` description drops `crons/` from the canonical-layout parenthetical.

**Update governance directive `query-handlers-read-only` (constitution + check.sh) to stop pointing at `crons/`.** The directive's check.sh logic is unchanged (still flags `db.exec(` / `.run(` inside `queries/*.js`). Only the human-facing strings update: the violation message changes from "move to actions/ or crons/" to "move to actions/", and the constitution body in both `CONSTITUTION.md` and `.governance/packs/srikanth235/centraid/directives/query-handlers-read-only/constitution.md` rewrites "Use `actions/*.js` (POST `/_run`) or `crons/*.js` for any writes" to drop the `crons/*.js` clause. The check.sh prose comment that explained which sub-paths are session-tracked is also updated.

**Remove `--dangerously-force-unsafe-install` from `scripts/reload-plugin.sh` (no more `openclaw cron` shell-out).** The unsafe-install flag existed solely because the plugin spawned `openclaw cron` via `child_process` at runtime, which OpenClaw's safe-install rejects. With the CLI shell-out gone, the flag and its explanatory comment are removed.

The deregister-cleanup fixtures, http-server bearer tests, and changes-sse SSE tests were all updated to drop the removed cron fields from their `RegistryEntry` literals and to drop the `NullScheduler` constructor arg from their `Runtime` setup; they continue to exercise the auth gate and the change-bus subscription model.
- **`bunx oxlint` (0 warnings/errors).** Linting runtime-core, openclaw-plugin, builder-harness, desktop, and mobile in isolation (the root `bun run lint` glob also pulls in a stale `.claude/worktrees/` checkout whose oxlint config is broken ‚Äî unrelated to this change) produces a clean run across 113 files / 364 rules.

## Out of scope

- **Bringing cron back behind a working backend.** The removal is deliberate; if scheduled execution lands later, it should arrive as one cohesive change with an executor that actually runs locally, not as a re-introduction of the partial wiring.
- **Migrating live `_registry.json` files.** Existing on-disk registries that still carry `cronTokens` / `cronStatus` keep loading ‚Äî extra fields are ignored and dropped on the next persist. No explicit migration shipped.
- **Cleaning historical receipts that mention cron.** Receipts (#38, #54, #56, etc.) reference the cron surface as it existed when those changes shipped. They're a historical record and stay as written.
- **Removing `crons/` directories from already-published app tarballs or templates on disk.** No template in this repo ships such files. Cleaning up user-side scaffolds is on the user.
- **Touching `bun.lock`'s transitive `croner` dependency.** It enters via `openclaw`'s own dependency tree, not centraid's. Removing it would require coordinating with the upstream openclaw package.
