# Three-tool dispatcher

## What's actually being dispatched

Each app's `queries[]` and `actions[]` arrays in `app.json` define **per-app handlers** — `set-cups`, `get-today`, `add-todo`, `complete-todo`, and so on. Those are what your UI calls and what an AI agent reasons about.

So why three generic tools and not "register every app's handlers as agent tools"? Because every time a new app is uploaded the catalog would grow, and every agent would need to relearn it. The dispatcher inverts that:

- **`centraid_describe`** — "what apps/handlers exist, with full schema". Discovery.
- **`centraid_read`** — "call query `<name>` with `<input>`". Generic read invocation.
- **`centraid_write`** — same, for actions (writes).

From the agent's perspective, three tools is all it sees registered; the per-app catalog is what it fetches via `centraid_describe`. The dispatcher is a _fan-out_ layer over the live handler catalog. There is also one built-in escape hatch, `_sql`, for ad-hoc reads/writes the app author never declared (see below).

The implementation is a single `Dispatcher` (`packages/app-engine/src/handlers/dispatcher.ts`), shared by all callers — the HTTP shim, the codex/Claude in-process agents, and the OpenClaw plugin — so the surface is byte-identical everywhere.

| Tool                | HTTP                                     | Purpose                                                  |
| ------------------- | ---------------------------------------- | -------------------------------------------------------- |
| `centraid_describe` | `POST /centraid/_tool/centraid_describe` | App manifest + live SQLite schema, or one handler entry. |
| `centraid_read`     | `POST /centraid/_tool/centraid_read`     | Invoke a query (or `_sql` SELECT).                       |
| `centraid_write`    | `POST /centraid/_tool/centraid_write`    | Invoke an action (or `_sql` write).                      |

The contract was added in issue [#107](https://github.com/srikanthsrungarapu/centraid/issues/107).

## Where `app` comes from

The dispatcher's request shapes carry an `app` field (`packages/app-engine/src/handlers/dispatcher.ts`):

- **Over HTTP** (`POST /centraid/_tool/<tool>`), the caller supplies `app` in the JSON body.
- **As an in-gateway agent tool** (codex / Claude SDK), `app` is bound automatically to the chat's scoped app (`ctx.appId`) — the model never passes it. See `packages/agent-runtime/src/backends/claude/host-tools.ts`.
- **As an OpenClaw agent tool**, the model passes `app`, and a `before_tool_call` hook cross-checks it against the session key (`centraid-conversation:<appId>`), refusing cross-app calls. `centraid_describe` with no `app` (list all apps) is the only allowed cross-app call. See `packages/openclaw-plugin/src/lib/tools.ts`.

## Response envelope

Internally the dispatcher returns an **MCP-shaped** `ToolResult`: `{ isError, content, structuredContent }`. `content` is a single `text` block carrying the JSON-stringified payload; `structuredContent` is the parsed payload (the success value, or `{ code, message, path? }` on error). Agents (codex / Claude / OpenClaw) see this shape directly.

The **HTTP shim** unwraps it (`packages/app-engine/src/runtime.ts`):

- **Success** → `200` with `structuredContent` as the raw JSON body (no wrapper).
- **Error** → status from `statusForToolError(code)`, body `{ code, message, path? }`.

See [Error codes](/reference/error-codes) for the full mapping.

## `centraid_describe`

Returns the app manifest plus the live SQLite schema, optionally filtered to a single action or query.

### Request

```json
{ "app": "hydrate", "action": "set-cups" }
```

All fields are optional:

- No fields → `{ apps: [{ id, manifest } | { id, error }] }` for every registered app.
- `app` only → `{ manifest, schema }` — the full manifest plus the live `data.sqlite` schema, in one round-trip.
- `app` + `action` → `{ app: { id, name, version }, action }` — just that action's manifest entry.
- `app` + `query` → `{ app: { id, name, version }, query }` — just that query's manifest entry.

### Response (HTTP success body, `app` only)

```json
{
  "manifest": {
    "id": "hydrate",
    "name": "Hydrate",
    "version": "0.1.0",
    "actions": [],
    "queries": []
  },
  "schema": { "schemaVersion": 1, "tables": [], "indexes": [], "views": [] }
}
```

## `centraid_read`

Invokes a query handler (read-only). For a declared query the dispatcher validates `input` against the query's `input` JSON Schema (Ajv) **before** the handler runs.

### Request

```json
{ "app": "hydrate", "query": "get-today", "input": {} }
```

| Field   | Required | Notes                                                                                                                          |
| ------- | -------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `app`   | yes\*    | Registered app id. \*Bound to the scoped app for in-gateway agent tools.                                                       |
| `query` | yes      | Name in the app's `queries[]`, or `_sql` for an ad-hoc SELECT.                                                                 |
| `input` | no       | Validated against the query's `input` schema. Omitted is treated as `{}`. For `_sql`: `{ sql: "<single SELECT or EXPLAIN>" }`. |

### Response

The handler's return value is passed through verbatim as the success body.

### `_sql` built-in (read)

`query: "_sql"`, `input: { sql }` runs one `SELECT` or `EXPLAIN` against the app's `data.sqlite`. Rows are capped at **200** (`SELECT_ROW_CAP`); DDL and PRAGMA are refused with `INVALID_INPUT`. Prefer a declared query when one fits.

### Errors

| Code                | Meaning                                                                            | HTTP shim status |
| ------------------- | ---------------------------------------------------------------------------------- | ---------------- |
| `UNKNOWN_APP`       | `app` doesn't match any registered app                                             | 404              |
| `UNKNOWN_QUERY`     | `query` doesn't exist on that app                                                  | 404              |
| `WRONG_KIND`        | A name exists, but it's an action, not a query                                     | 400              |
| `INVALID_INPUT`     | `input` failed schema validation (envelope includes `path`), or `_sql` was refused | 400              |
| `NO_ACTIVE_VERSION` | The app is registered but has no servable code                                     | 503              |
| `INVALID_MANIFEST`  | The app's `app.json` is malformed                                                  | 500              |
| `HANDLER_ERROR`     | Handler threw or timed out                                                         | 500              |

See [Error codes](/reference/error-codes) for the envelope shape.

## `centraid_write`

Invokes an action handler. Same validation; same dispatcher. The difference: a successful action fires the change bus for every table it touched, pushing events on `/centraid/<id>/_changes`.

### Request

```json
{ "app": "hydrate", "action": "set-cups", "input": { "cups": 5 } }
```

`action` may be `_sql` for an ad-hoc `INSERT`/`UPDATE`/`DELETE`/`REPLACE` (`input: { sql }`); DDL/PRAGMA refused.

### Response

Action handlers return `{ status?, body }`. The dispatcher **unwraps** this: the success body is `body`. If the handler returns a `status >= 400`, the dispatcher surfaces it as a `HANDLER_ERROR` instead of passing the error JSON through.

### Side effects

After a successful action, the runtime fires one `change` event per touched table on `/centraid/<id>/_changes`. Subscribed iframes re-fetch the affected queries. Tables an action mutates but doesn't declare in `writes: []` won't be invalidated — see [Actions → the change-bus contract](/build/actions#the-change-bus-contract).

## Calling from JavaScript

From inside an app iframe:

```js
async function readQuery(app, query, input) {
  const res = await fetch('/centraid/_tool/centraid_read', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ app, query, input }),
  });
  if (!res.ok) {
    const err = await res.json(); // { code, message, path? }
    throw new Error(`${err.code}: ${err.message}`);
  }
  return res.json(); // the handler's return value, unwrapped
}
```

(Subresource requests to the gateway are authenticated by the desktop's main-process auth-injector, which stamps the `Authorization: Bearer <token>` header onto gateway-origin requests.)

## Calling from an OpenClaw agent

On a remote gateway, the same three tools are registered with OpenClaw as agent tools (no HTTP needed):

```json
{
  "tools": {
    "profile": "coding",
    "alsoAllow": ["centraid_describe", "centraid_read", "centraid_write"]
  }
}
```

## Manifest validation

The dispatcher loads `app.json` and validates it when a handler is invoked. A malformed manifest (missing `id`, invalid JSON Schema in an `input`, etc.) yields `INVALID_MANIFEST`.
