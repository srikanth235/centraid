# Error codes

The dispatcher (`packages/app-engine/src/handlers/dispatcher.ts`) returns errors as **MCP-shaped** `ToolResult`s with `isError: true`. How that surfaces depends on the caller.

## What agents see (MCP shape)

In-process agents (codex / Claude SDK) and the OpenClaw plugin receive the raw `ToolResult`:

```json
{
  "isError": true,
  "content": [
    {
      "type": "text",
      "text": "{\"code\":\"INVALID_INPUT\",\"message\":\"...\",\"path\":\"/cups\"}"
    }
  ],
  "structuredContent": {
    "code": "INVALID_INPUT",
    "message": "input.cups must be a number",
    "path": "/cups"
  }
}
```

The agent backends collapse this to a single text block (`[CODE] message`) so the model reads the failure in its reasoning.

## What HTTP callers see (unwrapped)

The HTTP shim (`POST /centraid/_tool/<tool>`) does **not** forward the MCP envelope. On error it sets the status from the code and returns `structuredContent` directly (`packages/app-engine/src/runtime.ts`):

```json
{
  "code": "INVALID_INPUT",
  "message": "query \"get-today\" rejected input: must be number",
  "path": "/cups"
}
```

| Field     | Always present                                                       | Notes                                                 |
| --------- | -------------------------------------------------------------------- | ----------------------------------------------------- |
| `code`    | yes                                                                  | One of the codes below.                               |
| `message` | yes                                                                  | Human-readable explanation. Safe to surface to users. |
| `path`    | only for `INVALID_INPUT` (and only when a JSON Pointer is available) | JSON Pointer to the offending field in `input`.       |

(The shim's own framing errors — unknown tool name, non-JSON body — use a different shape, `{ error: <code>, message }`, since they never reach the dispatcher.)

## Code table

| Code                | Meaning                                                                         | HTTP shim status |
| ------------------- | ------------------------------------------------------------------------------- | ---------------- |
| `UNKNOWN_APP`       | `app` doesn't match any registered app                                          | `404`            |
| `UNKNOWN_ACTION`    | `action` doesn't exist on that app                                              | `404`            |
| `UNKNOWN_QUERY`     | `query` doesn't exist on that app                                               | `404`            |
| `WRONG_KIND`        | Name exists, but kind doesn't match (e.g. an action called via `centraid_read`) | `400`            |
| `INVALID_INPUT`     | `input` failed JSON Schema validation, or an `_sql` statement was refused       | `400`            |
| `NO_ACTIVE_VERSION` | App registered but has no servable code (no resolvable code dir)                | `503`            |
| `INVALID_MANIFEST`  | The app's `app.json` is malformed                                               | `500`            |
| `HANDLER_ERROR`     | Handler threw, timed out, returned `status >= 400`, or a built-in failed        | `500`            |

These eight codes are the `ToolErrorCode` union; the status mapping is `statusForToolError` in `dispatcher.ts`.

## When you'll see each

- **`UNKNOWN_APP`**: typo in `app`, or the app was deregistered between describe and read/write.
- **`UNKNOWN_ACTION` / `UNKNOWN_QUERY`**: typo in `action`/`query`, or the handler was removed from the live `app.json`.
- **`WRONG_KIND`**: called `centraid_read` with a name that's only declared as an action (or vice versa). The dispatcher knows the kind from `app.json`.
- **`INVALID_INPUT`**: the most common one for clients. Ajv-derived; `path` tells you which field. The handler did **not** run. Also covers `_sql` refusals (DDL/PRAGMA, or a write SQL on the read path).
- **`NO_ACTIVE_VERSION`**: the registry has the app but the code-dir resolver returns nothing servable (e.g. nothing published to the git store yet).
- **`INVALID_MANIFEST`**: `app.json` is broken or has an invalid `input` schema.
- **`HANDLER_ERROR`**: your handler threw, hit the worker timeout (10s for queries, 30s for actions — `packages/app-engine/src/handlers/dispatcher.ts`), or returned a non-2xx `status`.

## Catching from JavaScript

```js
const res = await fetch('/centraid/_tool/centraid_write', { ... });
if (!res.ok) {
  const err = await res.json(); // { code, message, path? }
  switch (err.code) {
    case 'INVALID_INPUT': showFieldError(err.path, err.message); break;
    case 'NO_ACTIVE_VERSION': showAppOfflineState(); break;
    default: showGenericError(err.message);
  }
  return;
}
const data = await res.json(); // unwrapped success payload
```
