# Governance directives

Centraid uses [`governance-kit`](https://github.com/srikanthsrungarapu/centraid/blob/main/CONSTITUTION.md) to enforce repository rules at commit time and in CI. Directives come from two packs under `.governance/packs/`: the upstream `governance-kit/core` pack (repo hygiene — commits, receipts, secrets, file size, docs) and the repo-local `srikanth235/centraid` pack (six Centraid-specific rules about app authoring and the gateway's architecture). Every directive has an executable `check.sh`; the whole layer runs through `.governance/run.sh`, invoked by the pre-commit hook locally and CI on every PR.

## The Centraid-specific directives

These six live in `.governance/packs/srikanth235/centraid/directives/`. Each `constitution.md` carries the full rule + rationale.

### `query-handlers-read-only`

**Rule.** A query handler (`*/queries/*.js`) must not mutate the database — no `stmt.run()`, no `db.exec()`. Writes go in `actions/*.js` (dispatched via `centraid_write`).

**Why.** The handler-runner skips SQLite session tracking on the read path as a perf optimization. A write from a query **succeeds silently** but is invisible to the change-notification feed at `/centraid/<id>/_changes`, so subscribed iframes never re-fetch — UI goes stale with no error.

**Waiver.** Per-line `// governance: allow-query-handlers-read-only <reason>` (e.g. lazy view materialization on first access).

### `actions-declare-table-writes`

**Rule.** Every entry in an `app.json#actions[]` array must include a `writes:` array of table names. `writes: []` is allowed and means "no DB writes" (e.g. a webhook-only action). Missing or non-array `writes` is rejected. Applies to any tracked `**/app.json` whose top-level `manifestVersion` is set (which excludes `apps/mobile/app.json`, an Expo config).

**Why.** Same foot-gun as `query-handlers-read-only`: the change-stream feed uses each action's declared `writes:` tables to invalidate query subscriptions. A missing/wrong `writes` field means the mutation succeeds but the bus stays quiet and the UI goes stale.

**Waiver.** None. The opt-out for a no-write action is the explicit empty array.

### `data-runtime-sqlite-separation`

**Rule.** Handler files (`**/queries/*.js`, `**/actions/*.js`) may not reference `runtime.sqlite` in any form. Handlers see only the app's `data.sqlite`, via the `ctx.db` proxy.

**Why.** Each app has two SQLite files. `data.sqlite` is app-owned (what `ctx.db` proxies). `runtime.sqlite` is gateway-owned — it holds the conversation ledger and automation state. A handler that opens or names `runtime.sqlite` is a layering violation the change-stream would never invalidate.

**Waiver.** Per-line `// governance: allow-data-runtime-sqlite-separation <reason>` (no legitimate case anticipated).

### `handler-uses-ctx-primitives`

**Rule.** Handlers must not import provider SDKs directly (`@anthropic-ai/sdk`, `openai`, `groq-sdk`, `@google/generative-ai`, `cohere-ai`, `@mistralai/mistralai`, `replicate`, `together-ai`). Inference and other gateway-managed capabilities flow through `ctx.infer.*` and related primitives.

**Why.** Handler-as-source-of-truth. Reaching past `ctx.*` defeats per-profile model routing, bypasses run-ledger cost accounting, and couples the handler to one provider — breaking the embedded ↔ OpenClaw gateway portability.

**Waiver.** Per-line `// governance: allow-handler-uses-ctx-primitives <reason>`.

### `no-hardcoded-model-ids`

**Rule.** Production source under `packages/` and `apps/` must not reference concrete provider model ids (`claude-opus-…`, `gpt-5`, `o1-mini`, `gemini-2.0-flash`, etc.) in string literals. Model selection flows through capability tiers (`tier:fast`, `tier:smart`) resolved at runtime. The single allowlisted file is `packages/app-engine/src/model-pricing.ts` (a price table is by definition a model-id map). Test files are excluded.

**Why.** Provider-agnostic inference. The model lineup churns; capability tiers abstract that churn behind a runtime resolver so model selection moves with operator preference, not code edits.

**Waiver.** Per-line `// governance: allow-no-hardcoded-model-ids <reason>` (e.g. a controlled experiment pinning a model).

### `gateway-engine-mode-agnostic`

**Rule.** Code under `packages/app-engine/` may not branch on which gateway mode it runs under. Mode-discrimination identifiers (`gatewayMode`, `gatewayKind`, `isEmbeddedGateway`, `isOpenClawGateway`, `isLocalGateway`, `isRemoteGateway`, `deploymentMode`, `hostingMode`, …) are forbidden in tracked source. Mode-specific behavior belongs at the entrypoints: `apps/desktop/src/main/` (embedded) and `packages/openclaw-plugin/src/` (OpenClaw).

**Why.** The "same code, two modes" property is what makes local-first-with-optional-remote cheap. Once `app-engine` checks which host it lives inside, dev and prod paths diverge.

**Waiver.** Per-line `// governance: allow-gateway-engine-mode-agnostic <reason>` (none today).

## Repo-wide directives that matter for app authors

From the `governance-kit/core` pack. These apply to anyone contributing to the repo (including app code that lands in `packages/blueprints/` or `apps/desktop/`):

| Directive                      | Summary                                                                                              | Waiver                                                     |
| ------------------------------ | ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| `repo-hygiene`                 | No merge markers, no build artifacts, no debug prints in non-test source, no source file > 500 lines | Line waiver `// governance: allow-repo-hygiene <reason>`   |
| `secrets-hygiene`              | No plaintext tokens or credentials in tracked files                                                  | Line waiver `# governance: allow-secrets-hygiene <reason>` |
| `commit-message-format`        | `<type>(scope)?!?: subject (#123)` with a trailing issue ref                                         | Merge/revert commits exempt                                |
| `no-orphan-todos`              | Every `TODO`/`FIXME` references a GitHub issue (e.g. `#120`)                                         | Line waiver                                                |
| `receipt-per-issue`            | Substantive work touches `receipts/issue-<N>-<slug>.md` with required sections                       | None — receipts are the system of record                   |
| `no-broken-internal-doc-links` | Every relative-path markdown link resolves                                                           | None                                                       |

The full core pack also includes `commit-issue-receipt-match`, `doc-freshness`, `required-docs`, `issue-templates`, `issues-tracked`, `agent-token-accounting`, `agent-steering-accounting`, and `workflows-hardened`. See [`CONSTITUTION.md`](https://github.com/srikanthsrungarapu/centraid/blob/main/CONSTITUTION.md) for the full text of each.

## Why directives live in the repo

The constitution's cardinal rule:

> Amendments to this constitution must land in the same commit as the change to its enforcing test. No exceptions.

A directive with no enforcing `check.sh` is a wish, not a rule. The whole layer is `.governance/run.sh` — the one script the pre-commit hook and CI invoke.
