# CLIs

Centraid ships two binaries:

- **`centraid`** (from `@centraid/agent-runtime`) — a thin shim over an app's `data.sqlite`. The local agent runtime shells out to it; you can use it directly to poke an app from a shell.
- **`centraid-gateway`** (from `@centraid/gateway`) — runs the gateway as a standalone daemon.

## `centraid`

The local agent runtime drives a turn through either the codex app-server (subprocess) or the Claude Agent SDK (in-process). Neither is OpenClaw, so neither has Centraid's tools registered natively. Rather than write engine-specific bindings, the runtime gives the engine one instruction: "to read or write app data, run `centraid sql …`". The CLI and the in-gateway agent tools share the same implementation in `@centraid/app-engine` (`sql-ops.ts`).

It operates on a SQLite file **relative to the current working directory** — there is no `--workspace` flag and no gateway/HTTP/token involved. The caller `cd`s into the app's data dir, or sets `CENTRAID_DATA_FILE` to an absolute path. Source: `packages/agent-runtime/src/cli/centraid-cli.ts`.

### Commands

| Command                                               | Purpose                                                                |
| ----------------------------------------------------- | ---------------------------------------------------------------------- |
| `centraid sql describe`                               | Print the schema of `data.sqlite`.                                     |
| `centraid sql read "SELECT …"`                        | Run a single `SELECT` / `EXPLAIN`; print rows as JSON.                 |
| `centraid sql write "INSERT/UPDATE/DELETE/REPLACE …"` | Run a single DML statement; print the result as JSON.                  |
| `centraid preview snapshot`                           | Report metadata for `.preview/snapshot.png` (path, exists, size, age). |
| `centraid --help` / `-h`                              | Usage.                                                                 |

DDL (`CREATE`/`ALTER`/`DROP`) and `PRAGMA` are **refused** in both `sql read` and `sql write`. `sql read` accepts only `SELECT`/`EXPLAIN`; `sql write` only `INSERT`/`UPDATE`/`DELETE`/`REPLACE`.

### Data file

The CLI reads `CENTRAID_DATA_FILE` if set; otherwise `data.sqlite` in the current working directory.

```sh
cd <app-data-dir>
centraid sql read "SELECT * FROM hydrate_daily"

# or
CENTRAID_DATA_FILE=/abs/path/data.sqlite centraid sql describe
```

### Output and exit codes

JSON on stdout for results; a short human summary on stderr. Exit codes:

| Code | Meaning                                             |
| ---- | --------------------------------------------------- |
| `0`  | success                                             |
| `1`  | runtime / SQL error                                 |
| `2`  | bad usage (missing arg, unknown subcommand)         |
| `64` | refused (e.g. a write passed to `sql read`, or DDL) |

## `centraid-gateway`

Runs the same gateway the desktop embeds as a standalone daemon. v0 scope (issue [#131](https://github.com/srikanthsrungarapu/centraid/issues/131)): loopback or LAN bind, no TLS, a single shared bearer token. Source: `packages/gateway/src/cli/cli.ts`.

### Subcommands

```sh
centraid-gateway serve [--config <path>] [--data-dir <path>] [--host <h>] [--port <p>]
centraid-gateway print-token --data-dir <path>
centraid-gateway --version
centraid-gateway --help
```

| Flag (`serve`) | Default     | Notes                                                            |
| -------------- | ----------- | ---------------------------------------------------------------- |
| `--config`     | —           | JSON config file. Either `--config` or `--data-dir` is required. |
| `--data-dir`   | —           | Gateway state root. Overrides the config file's `dataDir`.       |
| `--host`       | `127.0.0.1` | Bind host. Pass `0.0.0.0` for LAN reachability.                  |
| `--port`       | `0`         | `0` asks the OS for an ephemeral port. Must be in `[0, 65535]`.  |

`serve` flags override the config file. On boot the daemon mints (or reuses) a persistent bearer token at `<dataDir>/token.bin` (mode 0600) and prints the listening URL, the token, and the resolved `dataDir`. `SIGINT`/`SIGTERM` trigger a graceful shutdown. `print-token` reads back the persisted token (exit 1 if none exists yet).

There is **no TLS terminator** in v0 — front the daemon with Caddy / Tailscale Funnel / Cloudflare Tunnel if exposing beyond a trusted LAN.

### Config file

JSON; every field optional except `dataDir` (`packages/gateway/src/cli/config.ts`):

```json
{
  "dataDir": "/var/lib/centraid",
  "host": "0.0.0.0",
  "port": 8765,
  "runner": {
    "kind": "codex",
    "binPath": "/opt/homebrew/bin/codex",
    "extraArgs": ["--model", "<model-id>"]
  }
}
```

`runner.kind` is `"codex"` or `"claude-code"`. The `runner` block is seeded into the gateway's identity DB on first boot so the runtime's per-turn prefs loader picks it up.
