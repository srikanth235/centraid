# SQLite layout on disk

The gateway's state is files. There are **three** kinds of SQLite, each with its own migration ladder and connection (`packages/app-engine/src/stores/gateway-db.ts`, `packages/app-engine/src/insights/analytics-db.ts`):

- **Per-app `data.sqlite`** — handler-owned app data. One per app.
- **Per-app `runtime.sqlite`** — the conversation ledger + automation state. One per app.
- **Gateway identity DB** — `users` + `user_prefs`. One per gateway.
- **Gateway analytics DB** — one `run_summary` row per run. One per gateway.

App **code** is not on this list — it lives in the gateway's git store (issue #137), separate from all the state above.

> `@centraid/app-engine` is filename-agnostic: the host passes absolute paths for the identity and analytics DBs. The desktop names them `identity.sqlite` / `analytics.sqlite`; app-engine's own source comments use illustrative `centraid-gateway.sqlite` / `centraid-analytics.sqlite` labels, but the real filenames are whatever the host supplies.

## Desktop layout (per-gateway)

The desktop hosts one local gateway (fixed id `local`) plus 0..N remote gateways. Every file belonging to a gateway lives under one subtree (`apps/desktop/src/main/gateway-paths.ts`):

```
<userData>/gateways/<gatewayId>/
├── profile.json                  ← { id, kind, label, url?, createdAt }
├── token.bin                     ← encrypted bearer
├── identity.sqlite               ← users, user_prefs
├── analytics.sqlite              ← run_summary rollup
├── code-store/                   ← git store owning all app CODE (#137)
├── chat-runner-sessions/         ← codex/agent per-conversation scratch
├── templates-cache/, model-catalog.json, …
└── apps/                         ← per-app DATA storage
    ├── _registry.json
    └── <appId>/                  ← one per registered app
        ├── data.sqlite           ← app-owned (handler) data
        ├── runtime.sqlite        ← conversation ledger + automation state
        └── blobs/<sha256>        ← content-addressed attachment bytes
```

`rm -rf gateways/<gatewayId>/` removes everything about a gateway — no orphan files elsewhere. The standalone `centraid-gateway` daemon derives the same slots from its own root (`appsDir`, `identityDb`, `analyticsDb` are caller-supplied absolute paths — see `GatewayPaths`).

## `<appId>/data.sqlite` — handler data

The app's handler-owned database, at the app root (not in any code dir; code is versioned in git, data isn't).

- Persists across code changes and version swaps.
- Reachable only through query/action handlers (and the cloud-panel SQL routes). Never served as a static file, never uploadable, never opened by another app's worker.
- Reset by deleting the file and letting migrations re-run on next boot.

## `<appId>/runtime.sqlite` — the conversation ledger

The per-app conversation ledger + automation KV (`RUNTIME_MIGRATIONS` in `gateway-db.ts`). Runtime-owned; never reachable from handler `db` or the `centraid_*` agent tools. Five tables:

| Table              | What it holds                                                                                                                                                                                                           |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `conversations`    | The first-class durable thread. `kind ∈ {chat, build, automation}` lives **here**, not per-turn. Also carries `app_id`, `automation_id`, the `adapter_kind`/`adapter_session_id` resume handle, `turn_count`, `pinned`. |
| `turns`            | One execution under a conversation (a chat turn, an automation fire, a builder iteration). `conversation_id` is a NOT NULL, FK-backed, CASCADE spine. Carries the denormalized token/cost rollup written at finish.     |
| `items`            | The ordered trace, including the inbound message. `kind='message_in'` (ordinal 0) is the user/trigger input; `kind='step'` is one model-inference call (the token/cost grain); `tool`/`agent` are per-call audit rows.  |
| `attachments`      | Files riding an inbound message (chat upload OR webhook/email file), FK'd CASCADE to the `message_in` item. **Bytes are not here** — only metadata + the `hash`.                                                        |
| `automation_state` | Per-automation KV keyed by `(automation_id, key)`.                                                                                                                                                                      |

The data model is **conversation ⊃ turn ⊃ item**. There is no `run` / `run_nodes` layer — that was renamed and reshaped (issue #190). For `kind='automation'`, **each fire is its own conversation** (a fresh id tagged with `automation_id=<ref>`); an automation's run history is the set of conversations sharing that ref, and `automation_state` is the only thing that persists across them.

A turn's token/cost rollup (`total_*_tokens`, `total_cost_usd`, `step_count`, `tool_count`) is `Σ` over that turn's `kind IN ('step','agent')` items, computed at `finishTurn`. Per-call cost on each `step`/`agent` item is frozen at write time from the model price table (NULL when the model is unknown — distinct from a genuine $0).

`turns.conversation_id`, `items.turn_id`, and `attachments.item_id` are real same-file FKs (CASCADE) — deleting a conversation drops its turns, items, and attachment rows. `turns.parent_turn_id` stays a plain column: a cross-app sub-run (e.g. an `onFailure` cascade) has its parent in another app's file, and a SQLite FK can't span files.

## `<appId>/blobs/<hash>` — attachment bytes

Attachment bytes live content-addressed on the filesystem, keyed by the sha256 of their content (`data/blob-store.ts`) — never base64'd into a JSON column. Content addressing gives free dedup. The `attachments` rows carry the metadata + `hash`; GC is refcount-by-hash off `ConversationStore.referencedHashes`, run after a delete-conversation cascade.

## Gateway identity DB (`identity.sqlite`)

Two tables (`GATEWAY_MIGRATIONS`):

- `users` — the user identity row(s). Single-user today; the schema is multi-user-ready so a future shift doesn't need a column-add migration.
- `user_prefs` — global prefs keyed by `(user_id, key)`, JSON-encoded values, FK-cascaded from `users`.

## Gateway analytics DB (`analytics.sqlite`)

A denormalized `run_summary` table — one row per run (every kind, every app), the source the Insights screen reads (`ANALYTICS_MIGRATIONS`). At `finishTurn` the per-app ledger write-throughs a summary row here (sourcing `kind`/`app_id`/`automation_ref` from the owning conversation), so cross-app activity queries don't fan out across every app's `runtime.sqlite`. The write is best-effort — a failure never fails the run, and the per-app ledger stays authoritative.

## Pragmas + migrations

Every centraid SQLite file opens through the shared `openMigratedDb` with the same pragmas: `journal_mode=WAL`, `foreign_keys=ON`, `busy_timeout=30000`. Each file tracks its schema with `PRAGMA user_version`; the migrate runner applies the pending tail in one `BEGIN IMMEDIATE` and throws if the file is newer than the build understands. Pre-1.0, the baseline migration slot absorbs shape changes (no production rows to migrate); post-1.0 flips to strict append-only.

## What's NOT here

- **No `_chat/` directory, no per-window `.jsonl` transcript.** The transcript is the conversation's `items` rows, folded back on read — there is no separate transcript table or file.
- **No `run` / `run_nodes` tables.** Renamed to the conversation/turn/item model (issue #190).
- **No per-app user table.** Identity is the gateway-level `identity.sqlite`.
- **No cross-app handler database.** Each app's `data.sqlite` / `runtime.sqlite` are isolated; `analytics.sqlite` is a rollup handlers can't write to.

## Backing up

The whole `gateways/<gatewayId>/` subtree is the backup surface — the identity + analytics DBs, the git code store, and the per-app `apps/` data dirs (each with its `data.sqlite`, `runtime.sqlite`, and `blobs/`). Blobs are reconstructable only from re-uploads; the SQLite files are the state you can't reconstruct.

## Resetting an app

```sh
# Stop the gateway (close desktop, or stop the daemon)
rm <appsDir>/<appId>/data.sqlite     # clear handler-owned data
rm <appsDir>/<appId>/runtime.sqlite  # clear conversations + run ledger
rm -r <appsDir>/<appId>/blobs        # clear attachment bytes
# Start the gateway — migrations re-run on next boot
```
