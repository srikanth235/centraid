# Remote deploy — OpenClaw plugin

[`@centraid/openclaw-plugin`](https://github.com/srikanthsrungarapu/centraid/tree/main/packages/openclaw-plugin) mounts the Centraid gateway as a plugin on an existing OpenClaw instance. It's a thin shim over `@centraid/gateway`: the whole store / runtime / route graph is built by the same `buildGateway()` core the desktop embed and the standalone daemon use. OpenClaw provides the HTTP front door, auth, and process supervision.

This is the mode for "I want my apps reachable from my phone", "I want scheduled work to keep running", or "the family shares this gateway".

## What you get from OpenClaw

- A long-running HTTP gateway with TLS and auth.
- Process supervision — the plugin host restarts if it crashes.
- A standard plugin model so the same instance can host other extensions alongside Centraid.

What Centraid provides:

- The `/centraid` URL prefix and everything under it (app registry, per-app static + data + conversation surface, the three-tool HTTP shim, the template catalog).
- The sibling auth-bearing prefixes `/_centraid-conversations` (conversation turns + transcripts) and `/_centraid-user` (per-user store), plus the separately-mounted `/_centraid-hook` webhook route.
- The per-app DATA directory at `$OPENCLAW_STATE_DIR/centraid/` (default `~/.openclaw/centraid/`), with app CODE backed by a git store alongside it.
- The in-gateway agent tools (`centraid_describe`, `centraid_read`, `centraid_write`) for the OpenClaw-side conversation agent.

## Configure

The plugin reads exactly **one** config key at registration:

```json
{
  "appsDir": "centraid"
}
```

| Key       | Default                                                                  | Notes                                                                                                                |
| --------- | ------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------- |
| `appsDir` | `centraid` (resolved under `$OPENCLAW_STATE_DIR`, default `~/.openclaw`) | Where per-app DATA folders live. Absolute paths are used as-is; relative paths resolve under the OpenClaw state dir. |

The schema also declares a `versionRetention` key, but it's a holdover from the retired versioned-upload backend — the current plugin never reads it, so setting it has no effect. (See the [SQLite layout](/deploy/sqlite-layout) page for what ends up on disk.)

## State on disk

App **code** is backed by the gateway-owned git store (no tarball uploads, no version dirs). App **data** stays under `appsDir`. One level up from `appsDir`:

```
<state>/
  centraid/                     ← appsDir: per-app DATA (<id>/data.sqlite)
  centraid-code/                ← git store: apps.git + worktrees/ (app CODE)
  centraid-gateway.sqlite       ← identity (users + prefs)
  centraid-analytics.sqlite     ← central analytics
```

The runtime serves each app's handlers and static assets from the live `main` worktree, repointed atomically on every publish.

## Publishing an app

There is no tarball upload or version-flip step (both were retired with the move to the git store). App creation, template clone, and publish go through the gateway's lifecycle surface:

- `POST /centraid/_apps` — scaffold a blank app into a draft session.
- `POST /centraid/_apps/_clone` — clone a bundled template (see [Cloning a template](/templates/cloning)).
- The git-store session / files / publish routes commit a draft onto `main`.

Pass `{ "publish": true }` to scaffold or clone straight to live in one call.

## What's the same as local

Because both hosts mount the same `buildGateway()` core, the remote path is **not** a reduced one:

- **The builder works remotely.** Conversations (`kind ∈ {chat, build, automation}`) run through the same `/centraid/<id>/_turn` surface; the OpenClaw host injects an in-process conversation runner backed by `runEmbeddedAgent`.
- **The template gallery is gateway-owned.** `GET /centraid/_templates` resolves the bundle-or-cache catalog on the gateway, so a remote gateway serves the gallery too (and can fetch newer templates from a remote manifest URL).
