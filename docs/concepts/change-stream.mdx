# Change stream

The change stream is how a Centraid app stays live without polling. It's a single SSE endpoint per app that publishes one event for every successful write to the app's `data.sqlite`.

```
GET /centraid/<id>/_changes
```

Open a connection, get a stream of table-level invalidations whenever a write commits. The connection stays open until the client closes it.

## What lands on the wire

The handshake opens with a comment line so `EventSource.onopen` fires immediately, then every commit produces one `change` event:

```
: connected to <id>

event: change
data: {"tables":["todos"],"ts":1715812345678,"source":"handler"}
```

Fields on the JSON payload:

| Field        | Always | Meaning                                                                                                              |
| ------------ | ------ | -------------------------------------------------------------------------------------------------------------------- |
| `tables`     | yes    | The mutated table names ‚Äî sorted, deduplicated.                                                                      |
| `ts`         | yes    | Commit wall-clock time, ms since epoch.                                                                              |
| `source`     | yes    | `'handler'`, `'agent'`, or `'external'` ‚Äî who initiated the write (see below).                                       |
| `toolCallId` | no     | Present when `source === 'agent'` ‚Äî the CLI's tool-call id, so a renderer can pin a refresh to a specific chat pill. |
| `turnId`     | no     | Present when set by the agent path ‚Äî groups all writes from one turn.                                                |

A periodic heartbeat comment (`: ping\n\n`) goes out every 30s so proxies and browsers don't time the idle connection out. `X-Accel-Buffering: no` disables nginx response buffering for reverse-proxy deploys.

The serialization lives in [`changes-sse.ts`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/app-engine/src/http/changes-sse.ts); the payload shape is the `AppChange` interface in [`change-bus.ts`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/app-engine/src/changes/change-bus.ts).

## Why table-level, not row-level

Centraid's change bus deliberately publishes table invalidations rather than row diffs. The tradeoff:

- **Pro**: trivial to implement, trivial to subscribe to, works for arbitrary handler logic (the bus doesn't need to model your business rules).
- **Pro**: subscribers don't need to merge diffs into client state ‚Äî they just re-run the query.
- **Con**: noisier than row diffs. Every change to a 10k-row table wakes every subscriber on that table.

For the kinds of apps Centraid targets ‚Äî small, single-user, single-purpose ‚Äî the simpler model wins. A "todos" table doesn't get 10k rows.

## Where the invalidation comes from

Writes are observed by wrapping the SQLite connection in a `node:sqlite` session before the write and decoding the touched table names from its changeset after commit. `node:sqlite` doesn't expose the C `update_hook`, so this session-changeset approach is how the engine learns which tables moved. Several call sites participate, each tagging the `source`:

1. **`'handler'`** ‚Äî a user-authored **action** handler. The handler runner wraps each action's connection in a session and emits the touched tables after the handler returns success. ([`handler-runner.ts`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/app-engine/src/handlers/handler-runner.ts))
2. **`'agent'`** ‚Äî an in-process agent SQL write: a `centraid_write` call routed to the `_sql` built-in, emitted through the runtime's `agentEmitForApp` with the originating `toolCallId` / `turnId`.
3. **`'external'`** ‚Äî the cloud-panel SQL route (`POST /centraid/_apps/<id>/query`) and any other path without handler or agent context.

The bus is a synchronous, fire-and-forget, in-process pub/sub keyed by `appId` ([`ChangeBus`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/app-engine/src/changes/change-bus.ts)). It emits **after** the commit, so a subscriber never observes a change before it's durable. An empty table list is dropped ‚Äî read-only writers can call the bus without branching.

## What you can't write through

The `_sql` SQL ops (the raw-SQL escape hatch behind `centraid_read` / `centraid_write`) refuse DDL (`CREATE`, `ALTER`, `DROP`) and `PRAGMA`. Schema changes go through `migrations/`, not the runtime path. Action handlers run against a scoped DB proxy, which is the small surface that makes the invalidation tracking reliable.

## Subscribing from an app iframe

The pattern is plain `EventSource`:

```js
// inside app.js ‚Äî the iframe loads from /centraid/<id>/, so the
// relative path resolves to /centraid/<id>/_changes.
const es = new EventSource('_changes');
es.addEventListener('change', (e) => {
  const { tables } = JSON.parse(e.data);
  if (tables.includes('hydrate_daily')) refresh();
});
```

The event name is `change` and the payload field is `tables` ‚Äî both verified against the source above. The relative path works whether the gateway is local or remote.

## Why this matters for the read/write split

Queries are deliberately **not** tracked. The handler runner skips the session wrap for `handlerKind === 'query'` ‚Äî see the guard in `handler-runner.ts`. A `.run()` inside a query handler succeeds and mutates the database, but it never shows up on `_changes`. The iframe stays stale and no error fires. That silent class of bug is exactly what the read/write split exists to prevent ‚Äî see [Queries and actions](/concepts/queries-and-actions).
