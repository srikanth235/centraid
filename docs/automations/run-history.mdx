# Run history

Every time an automation fires, the runtime writes an audit trail to **`runtime.sqlite`** in the parent app's directory. This is separate from `data.sqlite` (the app's user data) — a clean line so automation churn doesn't pollute the app's data store.

```
<appDir>/
  data.sqlite           ← the app's user data (queries and actions touch this)
  runtime.sqlite        ← the automation ledger (the runtime writes this)
  versions/v_…/         ← code versions
```

The ledger follows one cohesive model — **`conversation ⊃ turn ⊃ item`** (the same spine that backs chat and build sessions). For automations specifically: **each fire is its own `conversations` row** (a fresh id, `kind: "automation"`, tagged with the automation ref in `automation_id`). The fire **is the turn** — there is exactly one `turns` row per conversation. Its `items` are the ordered trace (the inbound trigger plus one entry per `ctx.tool` / `ctx.agent` call). An automation's run history is the set of conversations sharing its ref — queried via `listAutomationTurns(ref)`.

> There is no `run` layer and no `node` layer. (Older docs saying `conversation ⊃ run ⊃ turn ⊃ node` are stale: the row that used to be a "run" is now the **turn**, and a "node" is now an **item**.)

## The schema

Three tables drive the ledger (DDL in `gateway-db.ts`; row types in [`packages/app-engine/src/conversation/schema.ts`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/app-engine/src/conversation/schema.ts)).

### `conversations`

One row per fire. `kind: "automation"`, `automation_id` = the automation ref (`<appId>/<id>`), `app_id` = the owning app.

### `turns`

One row per fire — the fire itself. Created when the trigger fires, finished when the handler returns (or throws).

| Column                    | Meaning                                                                                                                                                                |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `turn_id`                 | The run id. The handle for log lines, audit items, the run viewer. (`ctx.runs` surfaces it as `runId`.)                                                                |
| `conversation_id`         | NOT NULL FK into `conversations`, CASCADE. For an automation, the fire's own conversation.                                                                             |
| `trigger_kind`            | `"scheduled"`, `"manual"`, `"replay"`, `"on_failure"`, or `"interactive"` (chat) — the _intent_.                                                                       |
| `trigger_origin`          | `"cron"`, `"webhook"`, or `"manual"` — the _transport_.                                                                                                                |
| `ok`                      | Boolean. (`ctx.runs` filters surface this as `status: "ok" \| "error"`.)                                                                                               |
| `error`                   | When `ok` is false, the thrown message.                                                                                                                                |
| `summary`                 | The handler's returned `summary` string.                                                                                                                               |
| `output_json`             | The handler's returned `output`, JSON-serialized.                                                                                                                      |
| `started_at` / `ended_at` | Epoch ms.                                                                                                                                                              |
| `parent_turn_id`          | Set when this fire was a sub-run of another (e.g. an `onFailure` cascade).                                                                                             |
| token / cost rollups      | `total_input_tokens`, `total_output_tokens`, cache tokens, `total_cost_usd`, `step_count`, `tool_count` — Σ over the turn's `step` / `agent` items, written at finish. |

The trigger payload is **not** a turn column — it is the ordinal-0 `message_in` item (below).

### `items`

The ordered trace within a turn.

| Column                                    | Meaning                                                                                                    |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `item_id`                                 | Item id.                                                                                                   |
| `turn_id`                                 | FK into `turns`. CASCADE-deletes when the turn is pruned.                                                  |
| `ordinal`                                 | Position within the turn. Ordinal 0 is the inbound `message_in`; trace items start at 1.                   |
| `batch_id`                                | When multiple `ctx.tool` calls fire in one parallel batch, they share a batch id.                          |
| `kind`                                    | `"message_in"` (the trigger input), `"tool"`, `"agent"`, or `"step"` (a primary model inference).          |
| `name`                                    | Tool name, or `"agent"`. Absent for `message_in` / `step`.                                                 |
| `args_json` / `output_json`               | What was sent / what came back.                                                                            |
| `ok` / `error`                            | Outcome.                                                                                                   |
| `started_at` / `ended_at` / `duration_ms` | Timings.                                                                                                   |
| token / model fields                      | `input_tokens`, `output_tokens`, cache tokens, `model`, `provider`, `cost_usd` for `step` / `agent` items. |
| `child_turn_id`                           | The turn id of a child turn an `agent` item spawned. Dormant — no current producer.                        |

`items` is what the desktop's "Trace" view renders — every tool call and agent turn, in order, with timings. (Audit field values are capped at 64 KB; an oversize `args`/`output` is stored as a `{_truncated, bytes, head}` envelope.)

## Retention

`history.keep` on the manifest controls when old rows get pruned. Pruning runs **at end-of-run** on the automation that just finished — not on a global janitor — so it's lazy and bounded by how often the automation actually fires.

| `history.keep` shape | Behavior                                                                    |
| -------------------- | --------------------------------------------------------------------------- |
| `{ "count": N }`     | Keep the newest N runs. Default: `{ count: 100 }`.                          |
| `{ "days": N }`      | Drop runs whose `ended_at` is older than N days.                            |
| `"all"`              | Keep everything. No pruning.                                                |
| `"errors"`           | Keep only failed runs (`ok = 0`). Successful runs are pruned at end-of-run. |

Pruning removes the automation's `turns` (and their parent `conversations`); `items` cascade-delete with their `turns`.

**Practical guidance:**

- Low-frequency automations (weekly): `{ count: 100 }` keeps two years of history at no cost.
- High-frequency automations (every-15-minutes): `{ days: 30 }` keeps the audit useful while avoiding unbounded growth.
- Production failure-debugging: `"errors"` keeps only the rows you actually want to look at.
- The default (`{ count: 100 }`) is fine for almost everything; don't override unless you know you need to.

## How to read a run

### From the desktop

The per-app Automations screen lists every automation in the app. Click in to see the run list. Click a run to see:

- The trigger that fired it (cron, webhook, or "manual").
- The `summary` line.
- The full `input` (the `message_in` item) and `output` JSON.
- The trace — every `items` child, in order, with timings.
- The log output (lines from `log.info` / `log.warn` / `log.error` inside the handler).

### Programmatically, from another automation

`ctx.runs.last()` and `ctx.runs.list()` query the same ledger (via `listAutomationTurns`). See [Handler runtime → `ctx.runs`](/automations/handler#ctx-runs-prior-run-lookup).

### Directly, via SQL

The per-app run ledger is `<appsDir>/<appId>/runtime.sqlite` — at the app's **data** folder root, never inside a code-version directory (under the git-store backend, code resolves from a worktree but data stays at this stable path). `<appsDir>` is whatever root the host derives; the desktop and daemon supply it. To inspect:

```bash
sqlite3 <appsDir>/briefing/runtime.sqlite \
  "SELECT turn_id, ok, started_at, summary FROM turns ORDER BY started_at DESC LIMIT 20;"
```

## Per-run logs

`log.info(msg)` / `log.warn(msg)` / `log.error(msg)` calls inside the handler stream into a per-run log buffer (each takes a single string). The buffer is persisted to the automation app directory and shown in the desktop's run view.

Use them generously — they're the only way to see what's happening inside a handler that doesn't fail outright. (A handler that completes successfully but does the wrong thing leaves no `items` trace of the bug; the log is.)

## `parent_turn_id` and `onFailure` sub-runs

When a handler fails (throws, or a `ctx.tool` call rejects uncaught, or `outputSchema` validation fails) and `onFailure` is set on the manifest, the named automation fires as its **own** audited run:

1. A new fire (its own conversation + `turns` row) is created for the target, with `parent_turn_id` set to the failed run's `turn_id` and `trigger_kind: "on_failure"`.
2. The failure context (the failed run's id, automation name, and error) is passed in as `ctx.input` to the target's handler.
3. The target runs to completion (or failure) as its own fire, with its own `items` trace.

The target is a separate run, not nested inside the failed one — but `parent_turn_id` links the two. The `onFailure` handle resolves a bare `<id>` as a sibling within the same app. Cascades are capped at **depth 3** (the runtime refuses to push the chain further).

This is deliberate: the failure-handling automation should be auditable independently. You can look at the `onFailure` automation's run list and see every time it fired and what failure it was responding to (in its ordinal-0 `message_in` item).
