# Handler runtime

`handler.js` is plain JavaScript — a single ES module that default-exports an async function. It runs in a Node worker thread spawned by the runtime, with no DOM, no `db` proxy, and no access to other apps' data. Everything an automation can do flows through `ctx`.

The conventional filename is `handler.js`. The conventional location is `<appCodeDir>/automations/<id>/handler.js`. Both are constants in [`packages/automation/src/manifest/manifest.ts`](https://github.com/srikanthsrungarapu/centraid/blob/main/packages/automation/src/manifest/manifest.ts) (`HANDLER_FILE`, `MANIFEST_FILE`).

## The contract

```js
export default async ({ ctx, log }) => {
  // do work via ctx.* — the trigger input is ctx.input
  return {
    summary: 'one-line summary for the run list',
    output: {
      /* … */
    },
  };
};
```

The handler is invoked with a single argument object holding `ctx` and `log`. There is **no** top-level `input` argument — the trigger payload is read off `ctx.input`.

| Field       | What                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------- |
| `ctx`       | The runtime surface — see below.                                                                              |
| `log`       | `log.info`, `log.warn`, `log.error` — each takes a single string. Lines stream into the run's log buffer.     |
| `ctx.input` | The trigger's input payload. For webhook fires, the parsed HTTP body. For cron and manual fires, `undefined`. |

## Determinism — the replay-safe contract

A crashed fire **re-runs from the top** — there is no resume journal. So a handler must be deterministic, and all of its effects must go through the audited `ctx.*` rails (which are recorded in the run ledger and gated by `requires.tools`). An authoring-time lint (`lintHandlerSource` in `packages/automation/src/handler/lint.ts`) rejects the common offenders at publish time:

- **No ambient nondeterminism:** no `Date.now()`, no `new Date()` (no-arg), no `Math.random()`, no `randomUUID()` / crypto randomness, no `performance.now()`, no `process.env` / `process.cwd()` / `process.hrtime` / `process.argv`.
- **No raw I/O:** no `fetch()`, no `node:fs` / `child_process` / `net` / `http` / `https` imports. Reach external data through `ctx.tool(...)`.
- **Pure JS between `ctx.*` calls is free** — loops, conditionals, transforms.

Need "now" or a watermark? Derive it from `ctx.runs.last()` / `ctx.state`, or read a timestamp off a `ctx.tool` result — not the wall clock. (`new Date(value)` with an explicit argument is fine.)

**Return shape:**

| Field     | Notes                                                                                                                                                            |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `summary` | A short string. Shown in the desktop's run list as the one-liner under the timestamp. Omit it and the row shows `"ok"` / `"error"` only.                         |
| `output`  | Arbitrary JSON. Persisted on the turn (`outputJson`). If `outputSchema` is declared on the manifest, validated against it — a mismatch marks the run as `error`. |

`summary` and `output` are only picked up from a returned **object** (`{ summary?, output? }`). A returned bare string, number, or array is ignored.

A thrown exception (or a rejected Promise) marks the run as `error` with the thrown message. If `onFailure` is declared on the manifest, the sibling automation is fired with the failure context.

## `ctx.tool(name, args)`

Call an MCP tool. Returns the tool's result (or throws on failure — there is no built-in retry).

```js
const since = (await ctx.runs.last({ status: 'ok' }))?.startedAt ?? 0;
const events = await ctx.tool('google-calendar.list_events', {
  calendar_id: 'primary',
  time_min_ms: since, // derive the window deterministically, never new Date()
});
```

**The token-free rail.** `ctx.tool` is ~0 model tokens by construction: a long-lived agent session per fire is _puppeted by a mock-LLM server_, so the agent executes the staged tool through its native tool/MCP/auth machinery and returns the result without a real model ever being contacted. On local, that agent session is an in-process Claude Agent SDK turn or a `codex exec` subprocess; on the gateway it is an embedded agent run. All land at the same `ToolDispatcher` contract. The manifest's `requires.tools` scopes which tools the session is allowed to stage.

**Batching.** `Promise.all([ctx.tool(a), ctx.tool(b)])` collects the calls queued in the same microtask and dispatches them as one batch (one agent turn) — so parallel tool calls don't each pay a round-trip. Sequential `await ctx.tool(...)` calls are separate batches.

**Audit:** every `ctx.tool` call is recorded as an `items` row tagged `kind: "tool"` with the tool name, args, ok/error, result, and start/end timestamps. Calls in a parallel batch share a `batchId`.

**Retry is the handler's job.** There is no runtime retry. If a tool fails, the call rejects and the handler decides — `try/catch` and decide whether to retry, log, or fail the run.

## `ctx.agent({ prompt, json? })`

Run one constrained model turn. Returns the model's response. This is the **only billed rail** — use it solely for genuine inference (summarize / classify / extract / draft).

```js
const result = await ctx.agent({
  prompt: `Summarize these calendar events in one line:\n\n${JSON.stringify(events)}`,
  // Optional: a JSON schema. When present the host parses the answer and a
  // model failure to produce valid JSON is surfaced as an error.
  json: { type: 'object', properties: { summary: { type: 'string' } }, required: ['summary'] },
});
```

| Field    | What                                                                                                  |
| -------- | ----------------------------------------------------------------------------------------------------- |
| `prompt` | The natural-language prompt for the single turn.                                                      |
| `json`   | Optional. A JSON schema; when supplied the answer is parsed/validated and returned as a parsed value. |

**The model is picked by `requires.model`** on the manifest (`"anthropic/claude-3-5-sonnet"`, `"openai/gpt-4o"`, etc.); omitting it routes to the host's default automation model. The user's API credentials supply auth; the call is billed to their account.

**One turn, no tools.** `ctx.agent` is a _single_ model turn — no tool use, no multi-step loop. If you want the model to call tools, structure your handler as: call `ctx.tool` to gather data → call `ctx.agent` to reason over it → call `ctx.tool` to write the result back. The handler controls the loop; `ctx.agent` is a primitive inside it.

**Audit:** every `ctx.agent` call is recorded as an `items` row tagged `kind: "agent"` with the prompt, response, model id, and token counts when the provider reports them.

## `ctx.state` — cross-run KV

A per-automation key/value store that persists across runs. Backed by the `automation_state` table in the parent app's `runtime.sqlite`, keyed by the automation ref (`<appId>/<id>`).

```js
// A watermark cursor — read the last processed id, advance it deterministically.
const cursor = (await ctx.state.get('cursor')) ?? 0;
const items = await ctx.tool('example.list_items', { since: cursor });
if (!items.length) return { summary: 'nothing new' };
await ctx.state.set('cursor', items.at(-1).id);
```

| Method                      | What                                                                                                                        |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `ctx.state.get(key)`        | Returns the JSON-parsed value, or `undefined` if absent. Falls back to the raw string if the stored value isn't valid JSON. |
| `ctx.state.set(key, value)` | Persists the value (JSON-serialized). `undefined` becomes `null`.                                                           |
| `ctx.state.delete(key)`     | Removes the entry.                                                                                                          |

State is scoped to the automation ref. Two automations in the same app folder don't share state.

## `ctx.runs` — prior-run lookup

Read history of this automation (or another by ref).

```js
const last = await ctx.runs.last();
if (last && !last.ok) {
  log.warn('previous run failed, doing extra validation');
}

const recentFails = await ctx.runs.list({ status: 'error', limit: 10 });
```

| Method                   | What                                                                                                                                      |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `ctx.runs.last(filter?)` | Most recent prior run. Excludes the in-progress current run. Returns `undefined` if none. The filter accepts `automationId` and `status`. |
| `ctx.runs.list(filter?)` | Up to `filter.limit` (default 50) prior runs, newest first.                                                                               |

**Filter shape:**

| Field          | Notes                                                                         |
| -------------- | ----------------------------------------------------------------------------- |
| `automationId` | Defaults to the current automation's ref. Set to look at a sibling's history. |
| `status`       | `"ok"` or `"error"`.                                                          |
| `since`        | Epoch ms — only runs newer than this. (`list` only.)                          |
| `limit`        | Default 50. (`list` only.)                                                    |

Each returned ref carries `runId`, `automationId` (the ref), `triggerKind`, `startedAt`, `ok` (boolean), and — when present — `endedAt`, `error`, `summary`, `input`, and `output`. There is no `status` field on the returned ref; status is the `ok` boolean. (`status` is only a _filter_ key.)

## `onFailure` — declarative failure handoff

When the handler fails — it throws, a `ctx.tool` call rejects uncaught, or `outputSchema` validation fails — the runtime checks `manifest.onFailure`. If set, it fires that automation with the failure context as its `ctx.input`:

```json
{
  "name": "Daily PR digest",
  "onFailure": "notify-slack-on-failure",
  …
}
```

The named automation receives the failure info (`{ runId, automationName, error }`) as its `ctx.input` and can log to a table or post to Slack via `ctx.tool`. It runs as a separate audited fire (its own conversation + turn), linked to the failed one by `parent_turn_id`. A bare `<id>` resolves a sibling within the same app. Cascades are capped at depth 3. See [Run history → onFailure sub-runs](/automations/run-history#parent-turn-id-and-onfailure-sub-runs).

## A complete handler

A "weekly summary" automation that gathers data via `ctx.tool`, summarizes via `ctx.agent`, and stores the result via `ctx.tool` again. The "since" watermark comes from the prior successful run, not the wall clock, so a re-run after a crash stays deterministic:

```js
export default async ({ ctx, log }) => {
  // Deterministic watermark: the start time of the last successful run.
  const last = await ctx.runs.last({ status: 'ok' });
  const sinceMs = last?.startedAt ?? 0;

  const entries = await ctx.tool('centraid_read', {
    app: 'hydrate',
    query: 'list-since',
    input: { since: sinceMs },
  });

  if (!entries.rows.length) {
    return { summary: 'no entries this week, skipped' };
  }

  const result = await ctx.agent({
    prompt: `One short, kind line of encouragement based on this week:\n${JSON.stringify(entries.rows)}`,
    json: { type: 'object', properties: { text: { type: 'string' } }, required: ['text'] },
  });

  await ctx.tool('centraid_write', {
    app: 'hydrate',
    action: 'save-encouragement',
    input: { text: result.text },
  });

  log.info('encouragement written');
  return {
    summary: `wrote: ${result.text.slice(0, 50)}…`,
    output: { text: result.text, rowsConsidered: entries.rows.length },
  };
};
```

Note how the handler reaches its **own owning app's data through the three-tool dispatcher** (`centraid_read`, `centraid_write`) — not through a direct `db` proxy. Automations are clients of their app's tool catalog, same as a UI iframe.
