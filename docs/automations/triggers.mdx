# Triggers

`triggers` is an **array** of discriminated-union entries. Each entry has a `kind` field that picks which shape to expect: `"cron"` or `"webhook"`. An automation with no triggers — `triggers: []` or `triggers` omitted entirely — is legal: it fires only via an explicit "Run now" (or as an `onFailure` target).

```json
{
  "triggers": [
    { "kind": "cron", "expr": "0 9 * * 1-5" },
    { "kind": "cron", "expr": "0 17 * * 1-5" },
    { "kind": "webhook", "id": "a1b2c3…", "secretHash": "5f4d…" }
  ]
}
```

## Cron triggers

A `cron` trigger fires on a standard **5-field cron expression**:

```json
{ "kind": "cron", "expr": "<minute> <hour> <day-of-month> <month> <day-of-week>" }
```

| Field  | Allowed                                                                  |
| ------ | ------------------------------------------------------------------------ |
| `kind` | `"cron"`                                                                 |
| `expr` | A non-empty 5-field expression. Each field matches `[0-9*,\-/?A-Za-z]+`. |

The validator (`isValidCronExpression`) only checks field **shape** — five fields, each matching `[0-9*,\-/?A-Za-z]+`. The runtime matcher (`cronMatches`) supports numbers, ranges (`1-5`), lists (`1,15`), steps (`*/5`), wildcards (`*`/`?`), and Vixie day-of-month / day-of-week OR semantics (`dow` accepts both `0` and `7` for Sunday). It does **not** parse named days or months — `MON` / `JAN` pass the shape check but never match a minute, so the automation simply never fires (fail-safe). Use numeric fields.

**Time zone:** the in-process scheduler matches against the **local wall clock** of the machine running the gateway (matching the prior launchd/systemd behavior). There is no time-zone field in the manifest. `0 9 * * *` means 9am in the gateway host's local time.

**Multiple cron triggers are fine.** All of an automation's cron expressions are registered together; the scheduler fires the automation when any of them matches the current minute. A "weekdays at 9am AND 5pm" automation declares two entries, not one.

## Webhook triggers

A `webhook` trigger fires when an HTTP POST arrives at `/_centraid-hook/<slug>` on a **remote gateway**. The desktop preserves the entry but never registers the route — it is a gateway client, not an HTTP host. See [Webhooks](/automations/webhooks) for the route handler, auth flow, and rate limits.

### Provisioned shape

```json
{
  "kind": "webhook",
  "id": "a1b2c3d4e5f6...",
  "secretHash": "5f4dcc3b5aa765d61d8327deb882cf99..."
}
```

| Field        | Allowed                                                                                                                   |
| ------------ | ------------------------------------------------------------------------------------------------------------------------- |
| `kind`       | `"webhook"`                                                                                                               |
| `id`         | The route slug — matches `/^[A-Za-z0-9_-]+$/`. The runtime mints this from 12 random bytes (24-char hex).                 |
| `secretHash` | SHA-256 hex of the shared secret. The plaintext secret is shown to the user **once** at provisioning and never persisted. |

### Pending shape (builder handoff)

```json
{ "kind": "webhook", "pending": true }
```

When the user asks the conversational builder for a webhook trigger, the agent writes this shape — it cannot mint crypto-random credentials. The desktop's `provisionPendingWebhookAt` pass detects the pending entry on its next save, generates the `id` + secret, rewrites the trigger to its provisioned form, and surfaces the plaintext secret to the user once.

The validator accepts this shape so the manifest round-trips through the builder workflow. See [Conversational builder](/automations/conversational-builder) for the full handshake.

### At-most-one rule

`triggers` may contain **at most one webhook entry** (whether pending or provisioned). The validator throws `invalid_trigger` if you declare two. If you need fan-in from multiple endpoints, the pattern is one automation per endpoint, each writing to a shared store (`ctx.state` or a `centraid_write` table) that a downstream automation reads.

## Manual triggers — the empty array

`triggers: []` is legal and means "fire only when explicitly run." Two ways that happens:

1. **The "Run now" button** in the desktop's per-app Automations screen.
2. **As an `onFailure` target** — fired by the runtime when a sibling automation errors.

Useful for:

- Manual cleanup or report-generation tasks that shouldn't run on a clock.
- Targets of `onFailure` — a "log and notify" automation that fires only when its sibling errors.

## Legacy single-trigger shape

Older manifests carry a single `trigger` object (singular) instead of `triggers` (plural):

```json
{ "trigger": { "kind": "cron", "expr": "0 9 * * 1-5" } }
```

This shape is **dual-read** by `resolveTriggers`: it wraps the single object into a one-element `triggers` array. The manifest is rewritten plural the next time it's saved (by the builder, by the toggle, or by a webhook provisioning pass). You do not need to migrate by hand. New automations should use `triggers`.

## What happens when a trigger fires

Regardless of trigger kind, the firing path is the same:

1. The host (the gateway) resolves the automation by its handle.
2. If `enabled: false`, the fire is skipped (with a `200 OK / skipped: 'automation disabled'` response for webhooks; the in-process scheduler simply never registers a disabled automation).
3. The fire opens a fresh execution conversation + turn, then a worker spawns, loads `handler.js`, and invokes the default export. For webhook fires, the parsed HTTP body is passed in as `ctx.input`. For cron and manual fires, `ctx.input` is empty.
4. The handler runs. The trigger input, each `ctx.tool` and `ctx.agent` call, and the result are recorded in `runtime.sqlite`.
5. The returned `{ summary, output }` is persisted on the `turns` row.
6. Retention is applied per `history.keep`.

Two columns record how a fire was triggered. The turn's **`triggerOrigin`** is the transport — `"cron"`, `"webhook"`, or `"manual"`. The turn's **`triggerKind`** is the intent — `"scheduled"`, `"manual"`, `"replay"`, `"on_failure"`, or `"interactive"` (a chat turn). The desktop's run-list UI filters and groups on these.
