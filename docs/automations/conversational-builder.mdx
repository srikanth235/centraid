# Conversational builder

The conversational builder is the path most automations are created through: a chat surface where the user describes what they want, and the builder agent scaffolds the `automations/<id>/` directory — both the manifest and the handler.

The builder is the reason `prompt` lives on `automation.json` and `generated: {by, at}` is required. The manifest captures **what the user asked for**; `handler.js` captures **what the agent generated in response**. The pair round-trips through the builder: the next time the user says "now also do X," the agent re-reads both, plans a delta, and rewrites the handler.

Background: issue [#98](https://github.com/srikanthsrungarapu/centraid/issues/98) introduced the unified folder model + the chat-driven scaffolder. Issue [#91](https://github.com/srikanthsrungarapu/centraid/issues/91) is the earlier "automations as first-class projects" reframe the builder is built on.

## What the builder writes

For "Every Sunday at 8pm, summarize last week's hydration log and write a kind line of encouragement":

`automation.json`:

```json
{
  "name": "Weekly encouragement",
  "version": "0.1.0",
  "description": "A short, kind line of encouragement every Sunday evening, based on the week's hydration log.",
  "enabled": false,
  "prompt": "Every Sunday at 8pm, look at the past week's hydration log and write one short, kind line of encouragement based on how it went.",
  "triggers": [{ "kind": "cron", "expr": "0 20 * * 0" }],
  "requires": {
    "tools": ["centraid_read", "centraid_write"],
    "model": "anthropic/claude-3-5-sonnet"
  },
  "costEstimate": {
    "model": "anthropic/claude-3-5-sonnet",
    "tokensPerFire": 800
  },
  "history": { "keep": { "count": 100 } },
  "generated": { "by": "centraid-builder", "at": "2026-05-26T17:42:11.812Z" }
}
```

`handler.js` (deterministic — the "since" watermark comes from the last successful run, never the wall clock):

```js
export default async ({ ctx, log }) => {
  const last = await ctx.runs.last({ status: 'ok' });
  const since = last?.startedAt ?? 0;

  const week = await ctx.tool('centraid_read', {
    app: 'hydrate',
    query: 'list-since',
    input: { since },
  });

  if (!week.rows.length) {
    return { summary: 'no entries this week, skipped' };
  }

  const result = await ctx.agent({
    prompt: `One short, kind line of encouragement based on this week:\n${JSON.stringify(week.rows)}`,
    json: { type: 'object', properties: { text: { type: 'string' } }, required: ['text'] },
  });

  await ctx.tool('centraid_write', {
    app: 'hydrate',
    action: 'save-encouragement',
    input: { text: result.text },
  });

  log.info('encouragement saved');
  return { summary: `wrote: ${result.text.slice(0, 50)}…` };
};
```

The handler is plain JavaScript the user can edit. The agent's role is to scaffold an honest starting point, not to be the runtime — once it's written, `handler.js` is the source of truth and the agent re-reads it on the next iteration.

## Non-negotiable constraints

The builder agent is constrained — not by hand-wave, by validation. The runtime rejects manifests that violate these rules.

### `enabled: false` at scaffold time

The builder must write `enabled: false` on every freshly-scaffolded automation. A human turns it on. This is enforced by convention in the builder's system prompt, but the desktop UI also gates "first enable" behind an explicit click that surfaces the `costEstimate`.

Why: an automation that fires the moment the agent finishes typing is a footgun. The user should see the manifest, look at the handler, and decide.

### Never mint webhook credentials

When the user asks for a webhook trigger, the builder writes the **pending** shape:

```json
{ "triggers": [{ "kind": "webhook", "pending": true }] }
```

It does **not** invent an `id` or a `secretHash`. Those require:

- Crypto-random byte generation (an agent typing hex characters is not crypto-random).
- One-shot user surfacing (the plaintext secret has to be shown exactly once and never persisted).

The privileged server pass — `provisionPendingWebhookAt` — handles both. The builder's job is to declare intent; the runtime's job is to provision. See [Webhooks](/automations/webhooks#the-provisioning-handshake) for the full handshake.

### Never use the mock model

`requires.model = "centraid-mock/*"` would route `ctx.agent` calls back into the same mock provider the runtime tests use — an infinite-recursion footgun. The manifest validator throws `mock_model_disallowed` on any model id starting with `centraid-mock/`.

The builder agent picks a real model based on the user's available providers. If no provider is configured, the builder asks before scaffolding.

### Write a deterministic, replay-safe handler

A crashed fire re-runs from the top — there is no resume journal — so a handler must be deterministic and route all I/O through the audited `ctx.*` rails. An authoring-time lint (`lintHandlerSource` / `formatHandlerLintError`, surfaced by the gateway publish gate) statically rejects the common offenders — `Date.now()`, `new Date()` (no-arg), `Math.random()`, `randomUUID()`, raw `fetch`, `node:fs` / `child_process` imports, ambient `process.*`. The builder grounds on these rules so a handler is rejected at publish time, not at fire time. See [Handler runtime → Determinism](/automations/handler#determinism-the-replay-safe-contract).

### Always declare `generated`

The manifest validator requires `generated.by` + `generated.at`. The builder scaffold writes `"by": "centraid-builder"`; the shipped templates write `"by": "centraid-template"`; hand-authored files declare `"by": "hand"`. The field is provenance, not security — but it lets the desktop's "Automations" gallery flag scaffolded vs. hand-written entries.

## How the builder handles iteration

The user says: "Also CC me on Slack when you write the encouragement."

The agent:

1. **Re-reads `automation.json` + `handler.js`** in the existing directory.
2. **Plans a delta** — needs a Slack tool call after the `centraid_write`, so `slack.post_message` is added to `requires.tools` (the enforced `ctx.tool` allowlist).
3. **Rewrites both files** — manifest gets the new tool in `requires.tools`, handler gets the new `ctx.tool('slack.post_message', …)` call.
4. **Bumps `generated.at`** to the current ISO timestamp.
5. **Leaves `enabled` alone.** If the automation was enabled before the edit, it stays enabled; the user already approved the original.

The agent does not run a "build then test" loop in the user's chat. The handler is scaffolded statically — invoking it for real is a separate, audited action the user takes when they're ready.

## What the user can hand-edit

Anything. The builder is opinionated about its own output but doesn't lock the files. If the user opens `handler.js` in their editor and rewrites it, the next builder turn:

- Re-reads the (now hand-edited) handler.
- Plans the delta on top of the user's version.
- Preserves the user's structure where reasonable, but may rewrite if the user's ask requires a different shape.
- Rewrites `generated.by` to its builder value (overwriting `"hand"` if it was hand-edited) — provenance is per-write, not per-block.

The `prompt` field on the manifest can also be hand-edited — but it's documentation, not execution. Changing the prompt does not change the handler. To get the handler regenerated from a new prompt, ask the builder.
